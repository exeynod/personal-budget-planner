# Профилирование производительности — латентность + фронт (Home/Plan/Recurring)

Дата: 2026-06-10. Запрос владельца: «страницы грузятся долго», dev-стек, экраны Home/План/Recurring,
мерим латентность одиночных запросов + фронт. НЕ конкурентная нагрузка (личное single-owner приложение).

## Гипотеза

Латентность одиночного запроса (тяжёлые агрегаты), не конкуренция. Кандидаты:

- `GET /home` — главный агрегат дашборда (план/факт по категориям, баланс, карточки).
- `GET /periods/{id}/balance` — расчёт баланса.
- `GET /subscriptions/recurring/cashflow` — проекция кэшфлоу регулярок.
- `GET /plan-target`, `/planned`, `/subscriptions` — списки/агрегаты.
  Фронт: водопад зависимых fetch'ей (экран ждёт /periods → /home → ...), размер бандла, рендер.

## Целевые эндпоинты (из маппинга экранов)

| Экран     | Эндпоинты                                                                                   |
| --------- | ------------------------------------------------------------------------------------------- |
| Home      | `GET /home`, `/periods`, `/periods/{id}/balance`, `/planned`, `/subscriptions`, `/accounts` |
| Plan      | `/home`, `/plan-target`, `/periods`, `/planned`                                             |
| Recurring | `GET /subscriptions/recurring/cashflow`, `/subscriptions`                                   |

## Инфра

- **Dev-стек** (`$(DC)` = base+dev compose), DEV_MODE=true → initData обходится, эндпоинты доступны curl/httpx без токена.
- **Данные:** `make seed` (seed_extra_dev.py — богатые UAT-данные). При необходимости амплифицировать объём
  (больше транзакций/периодов), т.к. латентность агрегатов растёт с объёмом.
- Test-стек погасить (конфликт :8000).

## Метод (3 фазы)

### Фаза 1 — бэкенд-латентность (end-to-end)

- Харнес `scripts/perf/measure_latency.py` (httpx): каждый эндпоинт N=50 раз, cold+warm, p50/p95/p99/max.
- Выявить топ-медленные.

### Фаза 2 — атрибуция (app vs DB)

- Для медленных: получить SQL (SQLAlchemy echo / `pg_stat_statements` если есть) → `EXPLAIN (ANALYZE, BUFFERS)`.
- Искать: seq scan на больших таблицах, отсутствующие индексы, N+1 (много round-trip'ов), сортировки без индекса.

### Фаза 3 — фронт

- Playwright: загрузить Home/Plan/Recurring, снять network-водопад (тайминги, число запросов,
  последовательные vs параллельные), TTFB/LCP. Понять: водопад зависимых fetch'ей / бандл / рендер.

## Артефакты

- `scripts/perf/` — харнес(ы) (переиспользуемые).
- Отчёт `docs/perf/2026-06-10-latency-report.md`: таблица латентностей, EXPLAIN-выжимки, attribution,
  ранжированный список узких мест + конкретные фиксы (индекс/N+1/параллелизация фронта).

## Принципы

Сначала измерить, потом чинить (design-before-results). Фиксы — отдельным заходом после ревью отчёта.
Этот заход — ДИАГНОСТИКА: инфра + замеры + ранжированные узкие места. Код приложения не меняем (кроме
временного echo/таймингов, откат).
