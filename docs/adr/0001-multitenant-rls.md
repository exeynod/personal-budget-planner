# ADR-0001: Multi-tenancy через PostgreSQL RLS (per-row `user_id`, роли owner/member)

- **Статус**: Accepted
- **Дата**: 2026-06-07
- **Контекст**: CLAUDE.md (R9 / ARCH-A7), `app/db/session.py::set_tenant_scope`,
  миграции `0006_multitenancy`, `0007_postgres_role_split`, `0015_v10_rls_finalize`

## Контекст и проблема

Приложение задумывалось как личный планировщик одного владельца
(`OWNER_TG_ID`), но домен (категории, периоды, транзакции, подписки, AI) рано
или поздно должен обслуживать несколько пользователей (invite-флоу, роли). Без
изоляции один баг в `WHERE user_id = ?` приводит к утечке чужих данных.

Фильтрация по `user_id` руками в каждом запросе хрупкая: легко забыть условие в
новом сервисе/джобе/AI-tool, и это не ловится тестами автоматически. Нужна
изоляция, которая действует на уровне БД и не зависит от дисциплины в коде.

При этом рантайм-роль не должна иметь права обходить изоляцию, иначе RLS —
декоративный.

## Рассмотренные варианты

1. **Фильтрация в приложении (status quo)** — `WHERE user_id = :id` в каждом
   запросе. Просто, но не enforce'ится: забытое условие = утечка. Отклонено.
2. **Отдельная БД/схема на пользователя** — сильная изоляция, но дорогая
   эксплуатация и миграции для pet-app; оверкилл. Отклонено.
3. **PostgreSQL Row-Level Security + per-row `user_id` (выбран)** — политики на
   уровне БД, рантайм-роль без `BYPASSRLS`, scope ставится per-transaction.

## Решение

Доменные таблицы несут NOT NULL `user_id`. На них включён и форсирован RLS
(`0006_multitenancy`, добито `0015`). Политика:
`user_id = coalesce(NULLIF(current_setting('app.current_user_id', true),'')::bigint, -1)`.
Каждый запрос/джоба вызывает `set_tenant_scope` →
`set_config('app.current_user_id', <id>, is_local=true)` (transaction-scoped).
Рантайм работает под ролью `budget_app` (NOSUPERUSER **NOBYPASSRLS**), DDL —
под `budget` (`0007_postgres_role_split`). Роли пользователей —
`owner`/`member`/`revoked` (`UserRole`). `pdn_audit_log` намеренно вне RLS.

## Последствия

### Хорошие

- Изоляция enforce'ится БД, а не дисциплиной в коде — забытый фильтр даёт пустой
  результат, а не утечку.
- Один codebase обслуживает несколько тенантов; security-актив, а не carrying cost.

### Плохие

- Обязателен `set_tenant_scope` перед доменными запросами; забыл — «нет данных»
  (молчаливо, легко принять за пустую БД).
- Две роли БД усложняют локальную настройку и alembic.

### Нейтральные

- `SET LOCAL` транзакционен: сбрасывается на COMMIT/ROLLBACK — джобы ставят scope
  per-tenant в каждой итерации.
