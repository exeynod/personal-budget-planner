# ADR-0008: Экран месячного планирования (гейт перед тратами)

- **Статус**: Accepted
- **Дата**: 2026-06-08
- **Контекст**: ADR-0006 (планирование/UX), ADR-0007 (регулярные платежи),
  CLAUDE.md (период = Europe/Moscow, `period_for`, схема только через Alembic),
  модель `budget_period`, `GET /api/v1/home` (bootstrap), `NativeShell`.

## Контекст и проблема

Планирование месяца было «необязательным» экраном — пользователь мог сразу вносить
траты, не задав доход/лимиты/не разобрав регулярные. Владелец хочет сделать
планирование **ежемесячным ритуалом-гейтом**: на старте периода показывать отдельный
экран, который подводит «пора спланироваться» — провести то, что нужно сейчас, и
расписать план месяца. **Только после этого** открывается внесение трат.

## Решение

### Модель

- `budget_period.planned_at TIMESTAMPTZ NULL`. `NULL` = период не спланирован →
  показываем гейт. Заполняется при подтверждении.
- Миграция `0037`: добавить колонку (nullable), **бэкфилл** всех существующих
  периодов `planned_at = now()` — текущий незакрытый месяц НЕ гейтим на деплое.

### Триггер (авто + ручной)

- **Авто:** новые периоды из `close_period_job` создаются с `planned_at = NULL` →
  на первом входе в новый период показывается гейт.
- **Онбординг:** `create_first_period` ставит `planned_at = now()` — первый период
  считается спланированным (онбординг и есть планирование), двойного гейта нет.
- **Ручной:** кнопка «Спланировать» открывает тот же экран в любой момент (даже когда
  период уже спланирован) — без блокировки, с возможностью закрыть.

### Гейт — полный интерстишл

`NativeShell` при `needs_planning` рендерит `<PlanningGate>` ВМЕСТО обычного
таб-shell (`NativeChrome` + «+»/AddSheet). Экран перекрывает всё, пока не нажмёшь
«Готово».

### Завершение — явное «Готово»

Лёгкий критерий: пользователь проходит экран (видит due-платежи и план месяца) и
жмёт «Готово». `POST /api/v1/periods/{id}/confirm-plan` → `planned_at = now()` →
гейт снимается. Никаких обязательных под-шагов (можно расписать позже).

### Состав экрана

1. **«Провести сейчас»** — due/overdue регулярные платежи
   (`/recurring/due`) с действиями Оплачено/Пропустить/Перенести (как на главной).
2. **«План месяца»** — сводка: доход (Σ плановых зачислений), лимиты, «осталось
   распределить»; CTA в редактор плана/шаблона, чтобы расписать. Регулярные —
   через шаблон (ADR-0007).
3. **«Готово»** — подтверждение, снятие гейта.

### API

- `PeriodRead.planned_at: datetime | None`.
- `HomeResponse.needs_planning: bool` = `period is not None and period.planned_at is None`
  (чтобы shell гейтил на первом bootstrap без доп. запроса).
- `POST /api/v1/periods/{id}/confirm-plan` → ставит `planned_at = now()` (идемпотентно).

## Последствия

### Хорошие
- Ежемесячный ритуал: пользователь осознанно планирует до трат.
- Нет тихого «забыл задать план» — гейт подводит к процессу.
- Минимальная модель (один nullable-столбец + один confirm-эндпоинт).

### Плохие
- Полный интерстишл = жёсткая блокировка; если раздражает, ослабим до «только траты».
- `planned_at` живёт рядом с `closed_at`/`status` — ещё одно поле жизненного цикла периода.

### Нейтральные
- Бэкфилл существующих периодов как «спланированных» — гейт стартует со следующего.

## Сопутствующие фиксы (в этом же заходе)

- **Главная, карточка регулярных:** кнопки Оплачено/Пропустить/Перенести рисуются
  ПОД суммой платежа; наименование не обрезается (`RecurringDuePrompt`).
- **iPhone, клавиатура:** при фокусе на текстовом поле экран скроллит поле в видимую
  область (общий `NativeInput`/`useScrollIntoViewOnFocus`) — клавиатура больше не
  перекрывает поле.
