# ADR-0005: Period-движок `period_for` + расчёты в `Europe/Moscow`, БД в UTC

- **Статус**: Accepted
- **Дата**: 2026-06-07
- **Контекст**: `app/core/period.py::period_for`, `app/services/periods.py`,
  `app/worker/jobs/close_period.py`, CLAUDE.md (Conventions, TZ), docs/HLD.md §3

## Контекст и проблема

Бюджет ведётся не по календарному месяцу, а по циклу с настраиваемым днём
старта (`cycle_start_day`, напр. 5-е число). Нужно по любой дате однозначно
определять период `(period_start, period_end)` — для дашборда, агрегатов,
закрытия периода и проекции подписок.

Логика «какому периоду принадлежит дата» нетривиальна на краях месяца:
`cycle_start_day=31` в феврале, переход через декабрь/январь, инклюзивные
границы. Разбросать её по сервисам/джобам — гарантированные расхождения между
api и воркером.

Время и таймзоны: владелец и шедулер живут в МСК (джоба «закрыть период» в
00:01 должна срабатывать по МСК-полуночи), а БД должна хранить аудит-времена в
едином UTC, чтобы не зависеть от TZ-сервера.

## Рассмотренные варианты

1. **Календарный месяц (status quo)** — просто, но не соответствует домену
   (цикл стартует не 1-го). Отклонено.
2. **Ad-hoc вычисление периода в каждом месте** — расхождения api↔worker на
   краях месяца. Отклонено.
3. **Единая чистая функция `period_for` + явная политика TZ (выбран).**

## Решение

Одна чистая функция `period_for(d: date, cycle_start_day: int) -> (start, end)`
(`app/core/period.py`) — единственный источник истины о границах периода;
обе границы инклюзивны, `cycle_start_day` клампится к последнему дню месяца
(31→28/29 в феврале). Все расчёты периодов (api-сервисы и воркер) идут через
неё. Политика TZ: бизнес-даты — `DATE`; «сегодня» берётся в `APP_TZ`
(`Europe/Moscow`) через `_today_in_app_tz`; шедулер сконфигурирован на МСК;
аудит-времена — `TIMESTAMPTZ` в UTC.

## Последствия

### Хорошие

- Единая логика периодов — api и worker всегда согласованы на краях месяца.
- Корректные срабатывания cron по МСК при UTC-хранении в БД.

### Плохие

- Нужно держать в голове два «времени»: бизнес-даты в МСК vs аудит в UTC —
  легко перепутать при новом коде.
- `cycle_start_day` официально 1..28 (Pydantic `ge=1,le=28`); 29..31
  поддержаны лишь клампингом — скрытое ограничение контракта.

### Нейтральные

- `period_for` — чистая (без БД/IO), легко юнит-тестируется на edge-датах.

## Связи

- Реализовано в `app/core/period.py`, используется в periods-сервисах и
  `close_period`.
