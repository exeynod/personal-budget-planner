# ADR-0006: Перестройка планирования и UX (v1.2)

- **Статус**: Accepted
- **Дата**: 2026-06-08
- **Контекст**: CLAUDE.md (инварианты «деньги — копейки», «знак дельты»), миграции
  `0032_backfill_accounts`, `0033_drop_income_limits`, `0034_category_icon`,
  `0035_category_color`; коммиты `c6f0191`, `40e6846`, `bc87c3f`, `1eb81cb`,
  `3e226c8`, `b8f2567`, `bfb4d7e`; `docs/services/api.md`. Зафиксированы
  бизнес-решения серии итераций владельца по главной/плану/категориям.

## Контекст и проблема

После «planning rework» (миграции 0028–0031) часть продуктовой модели разъехалась
с UI: остаток на счёте брался из пустой таблицы `account`, у доходов существовал
«лимит», экран «шаблон плана» был удалён, доход за месяц нельзя было задать,
операции нельзя было редактировать, а главная грузилась водопадом запросов.
Владелец прошёл серию правок и зафиксировал ряд продуктовых решений, которые
дороже отменить, чем записать (затрагивают модель денег/плана и поведение для всех
пользователей).

Ключевые силы: единственный владелец + multi-tenant via RLS (решения применимы ко
всем пользователям); деньги — копейки; «положительная дельта = хорошо»; периоды —
Europe/Moscow. Бэкенд быстрый (~5мс), узкое место — RTT через Cloudflare-туннель,
поэтому число запросов на экран критично.

## Рассмотренные варианты

1. **Точечные правки без фиксации** — чинить по месту. Отклонён: решения
   противоречивы между собой и затрагивают модель — нужен журнал «почему так».
2. **Большой redesign модели плана за один заход** — отклонён как рискованный для
   живых данных; выбрана серия согласованных решений с миграциями данных.
3. **Зафиксировать принятые бизнес-правила (выбран)** — раскрыто ниже.

## Решение

Приняты следующие продуктовые правила (v1.2):

- **Кошелёк/остаток** = сумма балансов `account`; пользователям без счёта заводится
  один primary-счёт из текущего баланса периода (миграция 0032).
- **У доходов нет «лимита» как сущности** — только детализация плана (плановые
  доходные операции). Бэкенд отклоняет лимит на доходную категорию; легаси-лимиты
  доходов обнулены (миграция 0033). Доход месяца для «Осталось распределить»
  считается из Σ плановых зачислений, а не из отдельного поля настроек.
- **Подписки/«Регулярные платежи» — вне UX**: убраны из всех экранов; data-layer и
  джоба авто-списания (`close_period`) сохранены.
- **Шаблон месяца хранит и лимиты, и регулярные строки**; авто-применяется к
  КАЖДОМУ новому периоду (`close_period → apply_template_to_period`), текущий месяц
  не трогается. Шаблон можно перезаписать текущим планом из меню плана; строки
  шаблона планируются по числу месяца (1..27), а не по дате.
- **Иконка и цвет категории выбираются независимо** (как в iOS Shortcuts);
  `category.icon` (0034) + `category.color` (0035), оба nullable с fallback по имени.
- **Категории — пользовательские**: создание/переименование/архив + выбор
  иконки/цвета в Управлении.
- **Операции редактируемы**: дата, сумма, категория + удаление (PATCH/DELETE
  `/actual/{id}`).
- **Добавление операции учитывает вкладку** доход/расход (фильтр категорий по виду).

Enforcement: миграции (schema-SoT), `IncomeLimitForbiddenError` на бэке, ADR как
журнал. Деплой v1.2 — ручной на VPS из фичевой ветки (CI/авто-деплой осознанно
пропущены до мержа).

## Последствия

### Хорошие
- Один источник истины по остатку (счета) и по доходу месяца (план зачислений).
- Доходы и расходы не путаются (нет бессмысленного лимита дохода).
- Шаблон даёт повторяемый месячный план без ручного перевбивания.

### Плохие
- Миграции данных затрагивают всех пользователей (нужны бэкапы перед прогоном).
- `period_category_plan` и `category.plan_cents` сосуществуют (лимит пишется в оба
  при применении шаблона) — потенциальный дрейф, требует аккуратности.
- Подписочный UI удалён, но код data-layer остался (мёртвый до возврата UX).

### Нейтральные
- Появились новые обязательные поля иконки/цвета (nullable, с fallback).
- Шаблон использует число месяца 1..27 (не дату) — отдельная ментальная модель.
