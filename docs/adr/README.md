# Architecture Decision Records (ADR)

Записи архитектурных решений в формате
[MADR (Markdown Architecture Decision Records)](https://adr.github.io/madr/),
адаптированном под наш контекст: контекст и проблема, рассмотренные варианты,
решение, последствия (хорошие / плохие / нейтральные), связи. Тексты — на
русском; идентификаторы, команды и пути — на английском.

## Зачем

ADR — бортовой журнал «почему мы сделали так, а не иначе». README / HLD /
per-service доки описывают _что и как сейчас_; ADR фиксирует _какие
альтернативы рассматривались и почему выбрали эту_. Через полгода, когда
«кто-то предложит наоборот», ADR отвечает «вот почему мы так уже не делаем».

## Когда писать новый ADR

Новый ADR заводится, когда решение:

1. влияет на ≥2 сервиса или на длинный горизонт;
2. ограничивает будущие изменения (вводит запрет, gate, обязательный workflow);
3. дороже отменить, чем зафиксировать (миграция данных, breaking change, смена
   стека/дизайна);
4. требует обоснования в ревью больше двух раз — лучше написать один раз и
   ссылаться.

Мелкие решения (выбор библиотеки внутри одного модуля, рефакторинг,
форматирование) — _не_ предмет ADR. Сомневаешься — лучше короткий ADR, чем нет.

## Как нумеровать

Сквозная нумерация, четыре цифры: `0001`, `0002`, …; следующий ADR = максимальный
существующий +1. Номер не переиспользуется, даже если ADR отменён. Имя файла —
`NNNN-короткое-название.md` (kebab-case, английскими словами для поиска в IDE).

## Жизненный цикл статуса

- **Proposed** — на обсуждении.
- **Accepted** — принято, действует.
- **Deprecated** — больше не релевантно и не заменено другим ADR.
- **Superseded by ADR-XXXX** — заменён более поздним; перекрёстная ссылка
  ставится в обе стороны.
