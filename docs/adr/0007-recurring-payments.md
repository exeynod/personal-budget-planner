# ADR-0007: Регулярные платежи (обобщение подписок)

- **Статус**: Accepted
- **Дата**: 2026-06-08
- **Контекст**: ADR-0006 (планирование/UX v1.2), CLAUDE.md (инварианты «деньги —
  копейки», «soft delete только для category», «знак дельты», «схема — только через
  Alembic»), миграции `0028_planning_rework_add`, `0029_planned_posted_txn`,
  таблица `subscription` (Phase 22), джобы `close_period`, `charge_subscriptions`.
  Частично **заменяет** решения 0006 про «подписки вне UX» и «перезапись шаблона
  текущим планом».

## Контекст и проблема

В 0006 подписки были убраны из UX, но data-layer и джоба авто-списания сохранены как
«мёртвый код до возврата UX». Владелец хочет вернуть регулярные платежи, но не как
отдельную сущность «подписка», а как **обобщённый регулярный платёж** с произвольным
интервалом в месяцах (раз в N месяцев, день месяца), интегрированный в планирование.

Ключевая продуктовая граница, которую надо зафиксировать в модели:

- **Деталь шаблона** (`plan_template_line`, без даты) — повторяемая оценка «потрачу
  на психолога 4 раза, когда — неизвестно». Копируется в каждый период.
- **Регулярный платёж** — календарно-привязанное обязательство «2 числа раз в 2
  месяца». Имеет собственный график, может **пропускать** периоды — поэтому не
  выразим относительным `day_of_period` шаблона и должен быть отдельной сущностью.

Подписки (`monthly|yearly`) — частный случай регулярного платежа, отдельная сущность
больше не нужна.

## Рассмотренные варианты

1. **Оставить `subscription` + завести параллельно `recurring_payment`** — отклонён:
   две сущности об одном, дрейф логики, двойной UI.
2. **Переименовать таблицу `subscription` → `recurring_payment`** — чище семантически,
   но тянет миграцию FK (`planned_transaction.subscription_id`, partial-индексы,
   enum `plansource.subscription_auto`) на живых данных. Отклонён как дорогой/рисковый.
3. **Обобщить `subscription` in-place, ребрендить наружу (выбран)** — таблица и FK
   остаются, добавляем `interval_months`, мигрируем `cycle` → интервал, в API/UI
   называем «регулярный платёж». Раскрыто ниже.

## Решение

### Модель данных

- `subscription.interval_months` (SMALLINT, ≥1) — новый источник частоты:
  `1`=ежемесячно, `2`=раз в 2 мес, `12`=год. Бэкфилл из `cycle`
  (`monthly→1`, `yearly→12`); колонка `cycle`/enum `subcycle` помечается deprecated
  (не удаляем сразу — снижаем риск миграции). `day_of_month` (1..28) и
  `next_charge_date` (бегущий курсор) остаются якорем графика.
- Сдвиг курсора при материализации: `next_charge_date += interval_months` месяцев,
  день нормализуется к `day_of_month`.
- Хард-делит (как было; инвариант «soft delete только для category»).

### Три слота ввода (без пересечений)

| Где | Тип | Дата | Повтор | Хранение |
| --- | --- | --- | --- | --- |
| Шаблон | Регулярный платёж | график (день+интервал) | да, по календарю | `subscription` |
| Шаблон | Деталь (без даты) | нет | да, копия каждый период | `plan_template_line` |
| План периода | Разовая запланированная | дата опц. | нет | `planned_transaction(source=manual)` |

В шаблоне «+ строка» → развилка **Регулярный / Без даты**. В плане периода «+ строка»
→ только разовая. Регулярные заводятся только из шаблона/конфига, не из плана периода.

### Материализация и проведение

- Регулярные материализуются в период как **прогнозные**
  `planned_transaction(source=subscription_auto)` — учитываются в дельте план/факт, но
  **в факт сами не падают**.
- При создании посреди периода: если `next_charge_date` ∈ активный период — сразу
  материализуем в него (сейчас только на rollover).
- При открытии периода `close_period_job` материализует все попадающие платежи
  (механика есть; меняем шаг сдвига на `interval_months`).
- **Daily-джоба `charge_subscriptions_job` демонтируется** — «сегодня/просрочено»
  считается из уже материализованных строк, курсор сдвигается при материализации.

### Подсказка на главном (вместо авто-проведения)

- Карточка «Регулярные платежи»: **сегодня** + **просрочено** (`planned_date ≤ today`,
  не проведён).
- Действия: **Оплачено** (posting в факт, сумму можно поправить) / **Пропустить**
  (удалить это списание без факта) / **Перенести** (сдвинуть `planned_date` в пределах
  периода).

### Закрытие периода

- Непроведённые регулярные при закрытии **авто-пропускаются** (удаляются как
  «не случилось»). Cross-period overdue не вводим; риск гасится тем, что просрочка
  висит в подсказке весь период до закрытия.

### Отдельный экран — прогноз кэшфлоу (не список)

Список регулярных уже есть в шаблоне; отдельный экран даёт то, чего шаблон (разбитый
по категориям) не показывает — «когда и хватит ли денег»:
1. Таймлайн ближайших списаний (30–90 дней).
2. Проекция остатка по датам с подсветкой ухода в минус.
3. Месячная нагрузка (Σ/мес, % от дохода).

Заведение/правка — через шаблон; тап в таймлайне открывает тот же редактор.

### Отмена решений 0006

- **Удаляется** `save_template_from_current` (перезапись шаблона текущим планом) +
  роут + кнопка UI — признано ненадёжным. Шаблон правится только напрямую.
- Решение 0006 «подписки/регулярные вне UX» — отменено (возвращаем UX).

## Последствия

### Хорошие
- Одна сущность вместо «подписка + регулярный»; произвольный интервал в месяцах.
- Чёткая граница деталь-шаблона (без даты) vs регулярный платёж (по календарю).
- Нет тихого списания — пользователь подтверждает факт через подсказку.
- Экран кэшфлоу отвечает на вопрос, которого не было: «выдержит ли счёт график».

### Плохие
- `subscription.cycle`/`subcycle` остаётся deprecated-колонкой (дрейф; чистим позже).
- Авто-пропуск при закрытии может «проглотить» реальный платёж, если владелец
  игнорировал подсказку весь период.

### Нейтральные
- Внутреннее имя таблицы `subscription` расходится с внешним «регулярный платёж».
- `source=subscription_auto` сохраняется как значение enum (ребренд только в UI).
