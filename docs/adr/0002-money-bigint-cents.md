# ADR-0002: Деньги — `BIGINT` копейки, не float

- **Статус**: Accepted
- **Дата**: 2026-06-07
- **Контекст**: CLAUDE.md (Conventions), docs/HLD.md, поля `*_cents` в
  `app/db/models.py`, парсер `app/bot/parsers.py::parse_amount`

## Контекст и проблема

Приложение считает план/факт, дельты, остатки, проводит подписки. Денежные
ошибки округления недопустимы: дельта план−факт должна сходиться до копейки,
суммы агрегируются по категориям и периодам.

`float`/`double` для денег дают накопленную ошибку (0.1+0.2≠0.3) и
непредсказуемое округление при суммировании — классический баг финансовых
систем.

`NUMERIC`/`Decimal` точен, но тянет за собой разнобой типов между Python, SQL и
JSON и лишнюю сериализацию для простого домена «целые копейки».

## Рассмотренные варианты

1. **float/double (status quo по умолчанию)** — просто, но ошибки округления.
   Отклонено сразу.
2. **`NUMERIC(…)` / `Decimal`** — точно, но тяжелее в сериализации и типизации
   на всех слоях. Избыточно для целых копеек.
3. **`BIGINT` копейки (выбран)** — целочисленная арифметика, точная по
   определению; рубли — презентационный слой.

## Решение

Все денежные величины хранятся и передаются как `BIGINT` в копейках
(суффикс `*_cents`: `amount_cents`, `starting_balance_cents`,
`plan_cents`, …). Никаких float в денежных расчётах. Рубли получаются делением
на 100 только на UI. Парсер бота принимает «1500 / 1500,50 / 1 500₽» и
возвращает копейки (cap 10^12). Конвенция зафиксирована в CLAUDE.md и
enforce'ится в code review.

## Последствия

### Хорошие

- Точная целочисленная арифметика, дельты сходятся до копейки.
- Единый тип через все слои (БД/Python/JSON) — `int`.

### Плохие

- Любое отображение требует деления на 100 и форматирования (русская локаль:
  `1 500,50 ₽`).
- Внешние суммы (ввод пользователя, YooKassa) нужно конвертировать в копейки на
  границе.

### Нейтральные

- Суффикс `_cents` обязателен в именах полей — самодокументирующая конвенция.
