# Перф-отчёт: латентность Home/Plan/Recurring (2026-06-10)

**Запрос:** «страницы грузятся долго», dev-стек, экраны Home/План/Recurring, мерим латентность + фронт.
**Вывод одной строкой:** бэкенд НЕ виноват (p95 < 8ms на реалистичном объёме). Боль — **начальная
загрузка фронта** (парсинг JS-бандла + отсутствие route-сплита), доминирует на медленной сети/слабом CPU.

## Инфра (воспроизводимо)

- Dev-стек (`make up`), DEV_MODE → initData обходится. Прод-подобная отдача: **dev-caddy :80** (собранный
  SPA из `/srv/dist` + проксирование `/api`, `encode zstd gzip` включён).
- **Перф-данные:** `scripts/perf/seed_perf_data.py` — засеяно **25 периодов** (24 closed + 1 active),
  **5000** actual-транзакций (200/период), **15** подписок. Параметры: `PERF_MONTHS/PERF_TX_PER_PERIOD/PERF_SUBS`.
- Харнесы (переиспользуемые): `scripts/perf/measure_latency.py` (бэкенд), `scripts/perf/profile_frontend.mjs` (браузер, Playwright + CDP-троттлинг).

## 1. Бэкенд-латентность (N=40, warm, localhost)

ВСЕ целевые эндпоинты быстрые даже на 5000 транзакций / 25 периодов:

| endpoint                                | p50     | p95     | p99    | payload (decoded)              |
| --------------------------------------- | ------- | ------- | ------ | ------------------------------ |
| `GET /home`                             | 6.4     | 7.3     | 7.6    | **59.6 KB** (wire zstd ≈ 4 KB) |
| `GET /periods` (все 25)                 | 1.8     | 4.8     | 8.1    | 4.9 KB                         |
| `GET /periods/1/balance`                | 2.4     | 2.7     | 3.1    | 1.4 KB                         |
| `GET /periods/1/actual`                 | 3.4     | 4.4     | 6.7    | **50.8 KB**                    |
| `GET /subscriptions/recurring/cashflow` | 2.4     | 2.7     | 54.4\* | 6.9 KB                         |
| `GET /subscriptions`                    | 2.1     | 2.5     | 2.5    | 7.1 KB                         |
| analytics/\*                            | 2.2–3.5 | 2.4–5.2 | —      | < 1 KB                         |

\* единичный p99-выброс cashflow (87ms max), не систематический (p50/p95 = 2–3ms).

**Вывод:** оптимизировать SQL/индексы СЕЙЧАС бессмысленно — на этом объёме всё < 8ms. Бэкенд не
является причиной медленных страниц. (Замечание: `/home` и `/actual` отдают крупные decoded-payload'ы
59/51 KB, но по проводу zstd ужимает до единиц KB — на парсинг на клиенте всё же влияет.)

## 2. Фронт — начальная загрузка (Playwright, прод-бандл через caddy)

Реальный браузер (390×844 mobile), 3 профиля сети/CPU:

| Профиль             | wall (goto→idle) | FCP        | /home пришёл | DCL   |
| ------------------- | ---------------- | ---------- | ------------ | ----- |
| local (без троттла) | 661ms            | 88ms       | 142ms        | 55ms  |
| fast-4g + 4×CPU     | 1097ms           | 540ms      | 393ms        | 254ms |
| **slow-4g + 6×CPU** | **2101ms**       | **1412ms** | 1102ms       | 690ms |

- **12 запросов всего, 3 API** (`/home`, `/me`, `/subscriptions/recurring/due`) — идут **параллельно**
  (~одновременно), НЕ водопадом. Bootstrap-паттern `/home` (один round-trip отдаёт accounts+categories+
  period+actuals+planned и сидит кэши) работает — это хорошо.
- **JS:** decoded 460 KB (AppV10 270 + index 144 + dist 52), wire (gzip/zstd) ≈ **135 KB**. Сжатие ВКЛ.
- На slow-4g первый рендер (FCP) — **1.4с**, страница «устаканивается» к **2.1с**. Это и ощущается как «долго».

**Доминирующая стоимость** = загрузка ~135KB сжатого JS + **парсинг/исполнение 460KB decoded JS под
троттлингом CPU** до первого рендера. Не сеть-водопад, не бэкенд.

## 3. Переходы между экранами (Plan/Recurring)

Роутер стековый (`PosterRouter`), экраны **переиспользуют кэш** из bootstrap `/home` (`seedCache` в
`HomeMount.tsx`) → при навигации Plan/Recurring **нет повторных сетевых запросов**. Значит их «медленность»
(если ощущается) — это **стоимость React-рендера** под троттлингом CPU (длинные списки/гриды), а не сеть.
Прошлые (closed) периоды на Home идут granular-fallback'ом (~4 последовательных запроса) — вторично.

## Ранжированные узкие места + фиксы (по ROI)

1. **[ГЛАВНОЕ] Нет route-level code-splitting.** Чанк `AppV10` (270KB decoded / 70KB gz) тащит код ВСЕХ
   экранов (Home/Plan/Recurring/Analytics/Onboarding/Management) в первый рендер. → `React.lazy` + `Suspense`
   по экранам: первый paint Home не должен парсить код остальных экранов. Снижает и download, и parse на
   критическом пути. **Самый большой рычаг для «первая загрузка долгая».**
2. **Аудит бандла.** Прогнать `rollup-plugin-visualizer`/`vite build --analyze`: что в 270KB AppV10? Тяжёлые
   зависимости (графики аналитики, date-либы, иконки) — кандидаты на lazy-load/замену/tree-shake.
3. **Виртуализация длинных списков** (транзакции/категории) — режет render-стоимость под слабым CPU
   (адресует ощущаемую медленность Plan/Recurring, которые render-bound).
4. **Отложить некритичные API** — `/subscriptions/recurring/due` (и `/me`, если не нужен для первого
   рендера) запускать ПОСЛЕ first paint, а не в параллель на критическом пути.
5. **Granular-водопад для closed-периодов** — рассмотреть bootstrap-эндпоинт и для past-view (как у active),
   чтобы past Home не делал 4 последовательных round-trip'а. Вторично (active-view уже оптимизирован).

## Что НЕ надо делать

- Оптимизировать SQL/добавлять индексы под текущий объём — бэкенд уже < 8ms.
- Трогать сжатие — zstd/gzip уже включены во всех Caddyfile.

## Оговорки

- Замеры локальные (по запросу владельца). Прод-латентность API будет выше на величину сетевого RTT до VPS,
  но bootstrap-паттерн (1 параллельная пачка) это смягчает. Если боль сильнее локальных цифр — проверить
  прод отдельно (RTT до VPS, CPU реального устройства).
- LCP в headless не снялся (0) — использован FCP как прокси первого рендера.
- Профиль slow-4g+6×CPU намеренно жёсткий (mid-range телефон, посредственная сеть).

## Аудит бандла — полная композиция (sourcemap-анализ)

Сборка с `--sourcemap`, разбор `sources`/`sourcesContent` по пакетам (KB исходников = прокси вклада).

**Чанк `AppV10` (270KB min / 70KB gz) — 804 KB исходников, 199 модулей, ЕДИНЫЙ чанк всех экранов:**

|          KB (src) | Модуль                                                                   | Нужен для первого рендера Home? |
| ----------------: | ------------------------------------------------------------------------ | ------------------------------- |
|           **114** | `@phosphor-icons/react` (72 иконки)                                      | частично (Home — несколько)     |
|               111 | `screensV10/Management`                                                  | **НЕТ** (таб, push по тапу)     |
|                80 | `screensV10/native` (примитивы)                                          | да (shared)                     |
|                70 | `screensV10/Plan`                                                        | **НЕТ** (push из Home)          |
|                57 | `src/api`                                                                | да (shared)                     |
|                57 | `screensV10/Home`                                                        | да                              |
|                54 | `screensV10/AddSheet`                                                    | **НЕТ** (overlay по «+»)        |
|                40 | `screensV10/common`                                                      | да (shared)                     |
|                40 | `screensV10/Recurring`                                                   | **НЕТ** (push)                  |
| 31+31+25+25+19+18 | `Transactions`/`Analytics`/`Onboarding`/`CategoryDetail`/`Ai`/`Planning` | **НЕТ**                         |

**Чанк `dist` (52KB/17KB gz):** `valibot` **179 KB** + `@telegram-apps/sdk` 55 KB — тащится TG SDK, грузится на boot.
**Чанк `index` (144KB/47KB gz):** `react-dom` 130 KB — база, неизбежно.

**Архитектурный диагноз:** единственный сплит — `main.tsx → import('./AppV10')`. Дальше `AppV10 → NativeShell`
импортирует ВСЕ экраны **статически** (`React.lazy` = 0). Табы (`transactions/ai/management`) и detail-экраны
(`Plan/Recurring/CategoryDetail/...`) пушатся в роутер **по тапу**, но их КОД уже в критическом чанке.
Итого первый рендер Home парсит ~**400KB src** ненужного на старте кода + все 72 иконки.

**Рычаги (по убыванию вклада в критический чанк):**

1. `React.lazy` для табов (Management 111 + Ai 19 + Transactions 31 + Onboarding 25 = ~186 KB src вне Home-пути).
2. `React.lazy` для detail-экранов (Plan 70 + Recurring 40 + CategoryDetail 25 + Analytics 31 + AddSheet 54).
3. Иконки (114 KB): после per-screen сплита распределятся по lazy-чанкам.
4. `valibot` 179 KB (transitive через TG SDK) — проверить облегчение, ниже приоритет.

## Фикс №1 (code-splitting) — ПРОВЕРЕН И ОТКЛОНЁН (регресс)

Реализован `React.lazy` + `Suspense` для всех некритичных экранов (табы + detail + AddSheet). Статически
бандл сократился сильно: **AppV10 270.5KB → 38KB (−83%)**, появились lazy-чанки. НО замер «до/после»
(медиана 3 прогонов, slow-4g + 6×CPU, прод-бандл через caddy) показал **регресс first-paint:**

| slow-4g+6×CPU (медиана 3) | BEFORE (pre-split) | AFTER (split) | Δ                  |
| ------------------------- | ------------------ | ------------- | ------------------ |
| **FCP**                   | **1396 ms**        | **1688 ms**   | **+292 ms (хуже)** |
| load                      | 1053 ms            | 1248 ms       | +195 ms            |
| wall (goto→idle)          | 2107 ms            | 2614 ms       | +507 ms            |
| запросов на старте        | 12                 | 38            | +26                |

**Почему гипотеза не сработала:** вынесенные экраны (Management/Ai/Analytics/Transactions/...) и так
рендерились по тапу — их кода НЕ было на критическом пути ПЕРВОГО рендера Home. Сплит не уменьшил то, что
нужно Home, но **фрагментировал критический путь** (12→38 запросов): на 150ms-RTT сети лишние round-trip'ы
за Home-зависимостями (HomeMount/native-primitives/common, вынесенные в отдельные чанки) перевесили экономию
байтов. → **Сплит откатан** (`git checkout`), сохранён как `scripts/perf/rejected-codesplit.patch`.

## Скорректированная диагностика (по данным замера)

Ключевое наблюдение: **FCP (1396ms) > load (1053ms)**. Первый рендер происходит ПОСЛЕ загрузки ресурсов —
значит ~**340 ms** это **boot + render React под CPU-троттлингом**, а НЕ загрузка/парсинг бандла. Code-splitting
этот разрыв не лечит в принципе. Реальные узкие места первого рендера:

1. **CPU-стоимость рендера Home** (~340ms gap load→FCP под 6×CPU) — что именно рендерится на первом paint
   (тяжёлый дашборд? много category-row? heavy-эффекты?). → профилировать React render (Profiler API / what-renders).
2. **Критический download** (~1053ms load): `react-dom` (47KB gz, база) + **`valibot` 179KB src / `@telegram-apps/sdk`
   55KB** (грузится на boot в `dist`-чанке) + CSS 95KB. → trim TG-SDK/valibot — это крупнейшая НЕ-react
   критическая зависимость; проверить, можно ли инициализировать SDK легче/лениво.
3. **Boot-водопад**: `index.html → index.js → import('./AppV10') → Home`. Динамический хоп `main→AppV10`
   добавляет 1 RTT. → `modulepreload` критических чанков / убрать лишний динамический хоп.

## Рекомендации (скорректированные, в порядке вероятного ROI)

1. **Профилировать render Home** (React Profiler) — закрыть 340ms gap (мемоизация/виртуализация/дефер невидимого).
2. **Облегчить TG-SDK/valibot boot** (179KB) — ленивая/частичная инициализация SDK.
3. **`modulepreload` + убрать `main→AppV10` динамический хоп** — параллелить критический boot, минус RTT.
4. **НЕ** делать наивный route-split (доказанный регресс для first-paint на медленной сети).

## Что НЕ надо делать

- Оптимизировать SQL/индексы (бэкенд < 8ms).
- Трогать сжатие (zstd/gzip уже включены).
- Наивный code-splitting экранов (измеренный регресс first-paint).

## Фикс №2 (удаление @telegram-apps/sdk-react) — ПРИНЯТ (чистый, но скромный)

Приложение использовало SDK только для чтения initData (`retrieveLaunchParams`/`init`) + `openTelegramLink`,
а тащило весь SDK + **valibot 179KB** на boot-пути. Заменено на нативный `getInitDataRaw()` (читает
`window.Telegram.WebApp.initData` → URL-хеш `tgWebAppData` → sessionStorage-кэш — то же, что делал SDK, ~15 строк
без зависимости). `init()` убран; `openTelegramLink` → нативный. Файлы: `api/client.ts`, `main.tsx`, `package.json`,
2 теста (моки SDK убраны).

Результат: **чанк `dist` (valibot+SDK, 52KB/17KB gz) полностью исчез** из бандла. Замер (медиана 3, slow-4g+6×CPU):

|      | BEFORE  | AFTER №2    | Δ      |
| ---- | ------- | ----------- | ------ |
| FCP  | 1396 ms | **1300 ms** | −96 ms |
| load | 1053 ms | 957 ms      | −96 ms |
| wall | 2107 ms | 2017 ms     | −90 ms |

tsc ✅, vitest 209 ✅. **Принято** (в отличие от №1 — здесь нет регресса, только выигрыш). Но −7% — подтверждает,
что вес РАСПРЕДЕЛЁН, единичного фикса-серебряной-пули нет.

## Остаток и следующий рычаг

После №2 критический путь первого рендера: `index` (react-dom 47KB gz / 130KB decoded) + `AppV10` 70KB gz +
CSS 15KB gz, и **~340ms React-рендер** (FCP 1300 > load 957). Главный оставшийся рычаг — **react-dom → preact/compat**
(47KB gz → ~4KB gz + более быстрый рендер): потенциально −400-600ms. Это структурная замена с риском (compat-шимы,
тесты, phosphor/testing-library совместимость) — отдельное решение.

**Важная оговорка о масштабе:** «2с» — это намеренно жёсткий профиль slow-4g+6×CPU (mid-range телефон, плохая
сеть). Локально (как тестирует владелец) wall = **660ms**, fast-4g ≈ 1.1s. Если реальная боль = 2с, то устройство/
сеть владельца на медленном краю → preact + удержание веса. Если боль на проде — отдельно мерить RTT до VPS.
