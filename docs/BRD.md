# Бизнес-требования: TG Mini App «Бюджет-планировщик»

> **Статус:** v0.2 — первичное БТ + закрытые открытые вопросы (Q-1…Q-6, R-1…R-3).
> **Дата:** 2026-05-01
> **Источник:** `Google Monthly Budget 2026.02.xlsx` (личная таблица планирования бюджета).

---

## 1. Контекст и цель

Заказчик ведёт месячный бюджет в Google-таблице (5 листов: Summary, Plan, Transactions, Подписки, рабочий черновик). Цель — перенести этот же сценарий в Telegram Mini App, чтобы:

- вносить факт-расходы в один тап без открытия таблицы;
- видеть план/факт/дельту по категориям как в листе Summary;
- получать напоминания о списаниях подписок;
- сохранять «шаблон плана», который раскатывается на новые месяцы.

Это **личный pet-проект для одного пользователя** (single-tenant). Никакой публичной регистрации, биллинга и multi-tenant-изоляции в MVP не требуется.

## 2. Пользователь и роли

| Роль | Описание |
|---|---|
| Owner | Единственный пользователь. Авторизуется через Telegram (`initData`). Имеет полный доступ ко всем данным. |

Семейные категории («Здоровье Наташи», «Папа», «Выходы в вых. дни») **остаются обычными категориями расходов/доходов** — отдельной сущности «член семьи» нет.

## 3. Поверхности продукта

| Поверхность | Назначение | В MVP |
|---|---|---|
| **Telegram Mini App** | Основной UI: дашборд месяца, ввод транзакций, управление планом/категориями/подписками. | да |
| **Telegram-бот** (команды) | Быстрый ввод вне Mini App: `/add 1500 продукты пятёрочка`, `/balance`, `/today`. | да |
| **Push через бота** | Напоминания за N дней до списания подписки. | да |
| Веб-версия (вне TG) | — | нет |

## 4. Сущности и модель данных

Из таблицы извлечено 5 ключевых сущностей. Модель ниже — концептуальная (детальная схема — на этапе технического проектирования).

### 4.1 Category (Категория)
- `id`, `name`, `kind ∈ {expense, income}`, `is_archived`, `sort_order`
- Редактируется пользователем через UI (CRUD).
- Начальный набор НЕ импортируется из xlsx (старт с нуля), но в seed-фикстуре прикладываем дефолтный список из исходной таблицы как onboarding-предложение.
- **Архивация — мягкая:** архивированная категория исчезает из выпадающих списков при создании новых строк (план/факт/шаблон/подписки), но все исторические и текущие связанные записи остаются нетронутыми. Разархивировать можно в любой момент. Удаление категории через UI запрещено — только архивация.

### 4.2 PlanTemplate (Шаблон плана)
- Один на пользователя. Состоит из `PlanTemplateItem`.
- `PlanTemplateItem`: `category_id`, `amount`, `description`, `day_of_month?` (опц., если фиксированная дата списания).
- При создании нового месяца все элементы шаблона разворачиваются в `PlannedTransaction` этого месяца.

### 4.3 Month (Период бюджета)
- `id`, `period_start` (date), `period_end` (date), `starting_balance`, `ending_balance`, `status ∈ {active, closed}`.
- Период определяется глобальной настройкой `cycle_start_day` (default = **5**, редактируется в Settings). Пример: при `cycle_start_day=5` период «февраль» = 5 фев — 4 мар включительно. Изменение настройки применяется только к будущим периодам; уже закрытые не пересчитываются.
- Метка периода для UI: «месяц, на который приходится `period_start`» (5 фев → «Февраль 2026»).
- `starting_balance` нового периода = `ending_balance` предыдущего (автоматически при закрытии). Самый первый период — пользователь вводит `starting_balance` вручную в onboarding.
- `ending_balance` (факт) = `starting_balance + Σ income_actual − Σ expense_actual` (пересчитывается на лету).

### 4.4 PlannedTransaction (Плановая транзакция)
Соответствует листу **Plan**. Поля:
- `month_id`, `kind ∈ {expense, income}`, `amount`, `description`, `category_id`, `planned_date?`
- Источник создания: (a) разворот из `PlanTemplate` при старте месяца, (b) ручное добавление в UI, (c) автогенерация из `Subscription`.

### 4.5 ActualTransaction (Фактическая транзакция)
Соответствует листу **Transactions**. Поля:
- `month_id`, `kind`, `amount`, `description`, `category_id`, `date`, `source ∈ {mini_app, bot, subscription_auto}`
- Может быть «привязана» (опционально) к плановой строке для подсчёта дельты на уровне строки. В MVP агрегируем только на уровне категории.

### 4.6 Subscription (Подписка)
Соответствует листу **Мои подписки**. Поля:
- `name`, `amount`, `cycle ∈ {monthly, yearly}`, `next_charge_date`, `category_id`, `notify_days_before` (по умолчанию 2)
- Поведение:
  - **Push:** за `notify_days_before` дней бот отправляет напоминание.
  - **Автогенерация плана:** в день списания создаётся `PlannedTransaction` в текущем периоде на полную сумму (для годовых — полная годовая сумма в период списания, без размазывания).
  - После генерации плановой строки `next_charge_date` сдвигается на +1 месяц / +1 год.
  - **Дедупликация фактов не выполняется.** Если пользователь введёт факт-транзакцию вручную в день списания, появится дубль (план + ещё один план или план + факт-без-плана) — пользователь сам удаляет лишнее. В UI плановой строки от подписки выводим маркер «🔁 from subscription», чтобы было легко найти.
- Поле «Итоговая стоимость / мес» (1666 ₽ для годового Whoop) в MVP **не используется** — это была вспомогательная аналитика в xlsx, в продукте не воспроизводится.

## 5. Юзкейсы (MVP)

### UC-1. Просмотр сводки месяца (главный экран Mini App)
Эквивалент листа **Summary**. Содержит:
- Стартовый баланс, прогноз баланса на конец месяца, текущий баланс.
- Блок «Расходы»: план / факт / дельта в сумме и по каждой категории.
- Блок «Доходы»: план / факт / дельта в сумме и по каждой категории.
- Свитчер месяцев (←/→), переход в архивные закрытые месяцы — только чтение.

### UC-2. Добавить факт-транзакцию (Mini App)
Форма: дата (по умолчанию сегодня), сумма, описание, категория, тип (расход/доход). Категория — выбор из списка с поиском. После сохранения — мгновенное обновление сводки.

### UC-3. Добавить факт-транзакцию (бот)
Команды:
- `/add <сумма> <категория> [описание]` — расход
- `/income <сумма> <категория> [описание]` — доход
- Бот матчит категорию по подстроке без учёта регистра. При неоднозначности показывает inline-кнопки.
- Подтверждение в чат: «✓ 1500 ₽ → Продукты (Пятёрочка). Остаток по категории: 32 800 / 50 000».

### UC-4. Управление планом месяца
- Список плановых строк с группировкой по категории.
- Добавить / отредактировать / удалить строку плана текущего месяца.
- Кнопка «Применить шаблон» — заполнить пустой месяц из `PlanTemplate`.

### UC-5. Управление шаблоном плана
- Отдельный экран. CRUD строк шаблона.
- Кнопка «Перенести текущий план в шаблон» (на случай, если план месяца правился, и хочется зафиксировать его как новый дефолт).

### UC-6. Управление категориями
CRUD: добавить, переименовать, архивировать (с архивированной нельзя выбрать новую транзакцию, но историческая видимость остаётся).

### UC-7. Управление подписками
- Список (название, сумма, цикл, дата следующего списания, категория).
- CRUD.
- Тумблер «Уведомлять за N дней».

### UC-8. Push-напоминания о подписках
Шедулер ежедневно в 09:00 МСК сканирует подписки. Если до `next_charge_date` ровно `notify_days_before` дней — отправить push. В день списания — создать `PlannedTransaction` (если нет) и сдвинуть `next_charge_date`.

### UC-9. Закрытие периода / переход в новый
- В UI есть явная кнопка «Закрыть период» (или автозакрытие в день `cycle_start_day` в 00:01 МСК шедулером).
- При закрытии: фиксируется `ending_balance`, статус `closed`.
- При первом обращении к новому периоду (или в момент авто-перехода): создаётся новый `Month` с `period_start`/`period_end` по текущему `cycle_start_day`, `starting_balance = ending_balance` предыдущего, развёртывается шаблон плана.

### UC-10. Onboarding (первый запуск)
1. Пользователь открывает Mini App из бота → бэк проверяет `tg_user_id` против `OWNER_TG_ID`. Не совпадает → 403 «Доступ только владельцу».
2. Если совпадает, но в БД ещё нет `chat_id` (юзер открыл Mini App из ссылки, но не нажимал `/start` в боте) → onboarding-экран с кнопкой «Открыть бота» (`tg.openTelegramLink('https://t.me/<bot>?start=onboard')`).
3. После того как бот получил `/start` и сохранил `chat_id`, onboarding шаги:
   - ввод `starting_balance` для текущего периода;
   - подтверждение/изменение `cycle_start_day`;
   - предложение применить дефолтный seed категорий;
   - перевод на главный экран.

## 6. Бизнес-правила и расчёты

| Метрика | Формула |
|---|---|
| `expense_plan_total` | Σ `amount` по `PlannedTransaction` где `kind=expense` |
| `expense_actual_total` | Σ `amount` по `ActualTransaction` где `kind=expense` |
| `expense_delta` | **`expense_plan_total − expense_actual_total`** (>0 = экономия, зелёный) |
| `income_plan_total` | аналогично для `kind=income` |
| `income_actual_total` | аналогично для `kind=income` |
| `income_delta` | **`income_actual_total − income_plan_total`** (>0 = перевыполнение, зелёный) |
| `starting_balance` | переносится с прошлого периода, либо вводится вручную для первого периода |
| `ending_balance` (прогноз) | `starting_balance + income_plan_total − expense_plan_total` |
| `ending_balance` (факт) | `starting_balance + income_actual_total − expense_actual_total` |

**Правило знака дельты — единое:** «положительная дельта = всегда хорошо». Поэтому формула расходов и формула доходов разные. UI окрашивает положительную дельту в зелёный, отрицательную — в красный, ноль — нейтрально.

**Период считается по `cycle_start_day`:** транзакции с `date ∈ [period_start, period_end]` относятся к данному периоду. По умолчанию `cycle_start_day = 5` (т. е. период идёт с 5-го числа одного месяца по 4-е следующего). Меняется в Settings, влияет только на будущие периоды.

## 7. MVP-скоуп (что входит в первую версию)

✅ В MVP:
- Auth через Telegram WebApp `initData`.
- Категории (CRUD), только RUB, без мультивалютности.
- Месячный цикл с автопереносом баланса.
- PlanTemplate + развёртывание плана.
- Plan + Actual транзакции (CRUD).
- Сводка месяца (Summary как в xlsx).
- TG-бот команды: `/add`, `/income`, `/balance`, `/today`.
- Подписки + push за N дней + автогенерация плановой транзакции.
- Mini App: дашборд + 4 экрана (транзакции, план, категории, подписки).

❌ НЕ в MVP (post-MVP backlog):
- Графики/тренды по месяцам, алерты перерасхода.
- Импорт CSV/выписок банка.
- Импорт исходного xlsx.
- Многовалютность.
- Семейный учёт / роли.
- Веб-версия вне Telegram.
- Backup/restore через xlsx-выгрузку.
- Привязка ActualTransaction к конкретной строке плана.

## 8. Нефункциональные требования

| # | Требование |
|---|---|
| NFR-1 | Часовой пояс расчётов и шедулера — **Europe/Moscow**. Хранение времени в БД — UTC. |
| NFR-2 | Денежные суммы хранятся в **копейках** (integer) во избежание ошибок округления. На UI — рубли с двумя знаками. |
| NFR-3 | Telegram `initData` валидируется на каждом запросе по HMAC-SHA256 c `bot_token` (срок жизни — 24 ч). Whitelist: middleware сравнивает `tg_user_id` из валидной `initData` с ENV `OWNER_TG_ID`; всё, что не совпало, → 403. |
| NFR-4 | Все мутации идемпотентны там, где это возможно (особенно `subscription_auto` генерация плана — не должна дублироваться при повторном запуске шедулера). |
| NFR-5 | RPO = 1 сутки (ежедневный pg_dump). RTO — best-effort для pet-проекта. |
| NFR-6 | Целевая отзывчивость дашборда: < 300 мс TTFB при < 1000 транзакций/мес. |
| NFR-7 | UI следует Telegram Theme (`tg.themeParams`), поддерживает тёмную тему. |

## 9. Технические рамки (зафиксировано опросом)

| Слой | Решение |
|---|---|
| Backend | Python, FastAPI + SQLAlchemy, Pydantic v2 |
| TG-бот | aiogram (async). В отдельном контейнере `bot` — он же отправляет push-сообщения, дёргая внутренний эндпоинт API. |
| БД | PostgreSQL 16 |
| Frontend Mini App | **React 18 + Vite + TypeScript + `@telegram-apps/sdk-react`**. UI-kit — на этапе HLD (кандидаты: `@telegram-apps/telegram-ui`, shadcn/ui, собственный) |
| Шедулер-процесс | **Отдельный контейнер `worker`** в docker-compose (APScheduler в собственном процессе). Делит БД с API. Для координации с API — PostgreSQL advisory locks. |
| Хостинг | VPS (Hetzner / Timeweb / Yandex Cloud), docker-compose |
| CI/CD | GitHub Actions → деплой по push в `main` |
| Реверс-прокси | Caddy / nginx с автоматическим Let's Encrypt |
