# Liquid Glass v2 — UX-планирование + Liquid Glass: ресёрч для согласования

> Цель документа: согласовать с владельцем (а) исправленный UX-сценарий планирования (лимиты → детализация → «провести в факт») и (б) апгрейд визуала Liquid Glass. Ниже — как есть сейчас, где разрыв, что предлагается, что переиспользуем и какие решения нужны от владельца.

---

## 1. Текущий UX-путь планирования (как есть)

Веб-клиент — это шелл `liquid_glass` (native), экраны в `frontend/src/screensV10`. Что реально построено:

**(a) Установить лимит по категории — РАБОТАЕТ.**

- Путь: Home → строка **«План месяца»** → редактор Plan → инлайн-поле ₽ на каждой категории → **«Сохранить»**.
- Контрол: нативное числовое поле (`native-plan-input-{id}`), НЕ слайдер (слайдер только в poster-шелле). Ввод рублей → `rublesInputToCents` → правит локальный драфт (без PATCH на каждое нажатие).
- Карточка сверху **«Осталось распределить»** = `income − Σplan`. Если отрицательно → красный бейдж «Превышено» и **Save заблокирован**.
- **«Сохранить»** → `PATCH /api/v1/plan-month` `{plans:[{category_id, plan_cents}]}`, атомарно, сервер валидирует `Σplan ≤ income` (400 `plan_overflow`). Лимит хранится в `Category.plan_cents` — единое **живое** значение, НЕ per-period, НЕ шаблон.

**(b) Добавить транзакцию-детализацию внутри лимита — НЕ ПОСТРОЕНО в вебе.**

- В редакторе Plan на строку категории есть только одно число (лимит) + тумблер rollover. Нет способа добавить line-item внутрь лимита.
- Бэкенд это **полностью умеет** (`GET/POST /periods/{id}/planned`, `PATCH/DELETE /planned/{id}`), типы (`PlannedRead`) сгенерированы, **но нет рантайм-обёртки в `api/v10/` и ни один экран их не вызывает.**
- Единственное «planned»-подобное в Plan — секция **«Регулярные · провести в факт»**, но это **подписки** (`subscription`), создаваемые на отдельном экране, а не детализация внутри лимита.

**(c) «Провести в факт» — частично, только для подписок.**

- В Plan секция «Регулярные»: на каждой подписке кнопка **«Провести»** → `POST /api/v1/subscriptions/{id}/post`. Бэкенд вставляет `actual_transaction` и ставит `subscription.posted_txn_id`. Кнопка флипается в «Отмена» (`unpost`).
- Ограничения: **дата зашита на «сегодня»** (нет выбора дня), **по одной** (нет «провести всё»), **только подписки** (ручные planned провести нельзя — их негде создать). При отсутствии подписок секция пустая/мёртвая.

**(d) Записать реальную трату (факт) — РАБОТАЕТ полностью.**

- «+» (на Home или через `AddSheetHost`) → **AddSheet**: крупная сумма + нативный кейпад, описание, инсет-строки Категория / Счёт / Дата (Сегодня / Вчера / Своя дата → скрытый `<input type=date>`).
- Submit → `createActualV10({kind:'expense', amount_cents, description, category_id, tx_date, account_id})`. Это и есть «в один тап записать факт» из Core Value. Только расход (без дохода).

## 2. Связь план↔факт сегодня (и где разрыв)

Связь существует **только на чтении/агрегации, никогда на вводе**.

**Что связано (read-side):**

- Движок — `compute_balance` (`app/services/actual.py:378`). План-сторона = `Category.plan_cents` по активным **расходным** категориям (исключая `code='savings'`); income-план = единое `AppUser.income_cents`. Факт-сторона = `SUM(abs(actual_transaction.amount_cents))` по `(category_id, kind)`.
- Дельта: расход `план − факт`, доход `факт − план`.
- На UI Home/CategoryDetail показывают **План / Факт / В запасе** + прогресс-бар. То есть «видеть дельту» работает.

**Где разрыв (три точки):**

1. **`planned_transaction` полностью исключён из плана.** Докстринг `compute_balance` прямо это фиксирует: канонический план v1.0 живёт в `Category.plan_cents`, а planned-строки суммировать нельзя — это и пропустит реальный план, и **задвоит подписки** (их авто-чарджи уже лежат как `subscription_auto`). Итог: детализация, которую владелец хочет вводить, **невидима для бюджетной математики**.
2. **Нет связи detail → actual.** Ни `posted_txn_id`/`is_posted` на `planned_transaction`, ни `planned_id` на `actual_transaction`. Система не знает, какие planned-строки реализованы; нельзя посчитать «лимит − запланировано − уже потрачено».
3. **Лимит — глобально-живой, не per-period.** `Category.plan_cents` — одно мутабельное значение; закрытие периода его не снапшотит.

**Одной строкой:** лимит (`Category.plan_cents`) и детализация (`planned_transaction`) уже существуют, но живут в двух параллельных вселенных; мост (`posted_txn_id`, как у подписок) есть только у подписок, а веб-UI к planned-строкам не прикасается вообще.

## 3. Предлагаемый исправленный UX-сценарий

Принцип, согласованный с интентом владельца: **Лимит = потолок. Детализация = опциональные line-items внутри потолка. Проведение = превращение запланированной строки в реальную трату на выбранную дату.** План и факт связываются на записи через новый мост `posted_txn_id`.

### Шаг 0 — Войти в «режим планирования»

- Home → строка **«План месяца»** → редактор Plan (существующий). Отдельная точка входа не нужна — Plan и есть режим планирования, мы его расширяем.
- Читает: сводку периода + `Category.plan_cents` (есть). НОВОЕ: per-category агрегат `Σ planned (manual, unposted)`.

### Шаг 1 — Лимиты по категориям (уже построено, оставляем)

- Инлайн-поле ₽ → `PATCH /api/v1/plan-month`, атомарно, `Σplan ≤ income`, карточка «Осталось распределить», overflow → Save заблокирован. Без изменений, кроме добавления affordance детализации (Шаг 2).

### Шаг 2 — Детализация внутри лимита (НОВОЕ)

- На строке категории — disclosure/chevron → инлайн-секция (или под-экран) **«Детализация лимита»** для этой категории.
- Внутри:
  - Список planned line-items (`GET /periods/{id}/planned` отфильтрованный по `category_id`, `source=manual`). Строка: описание, сумма, опц. дата, кнопка **«Провести»** (Шаг 3).
  - **«+ Добавить запланированную трату»** → форма (описание, сумма ₽, опц. дата) → `POST /periods/{id}/planned`.
  - Edit/delete → `PATCH`/`DELETE /planned/{id}`.
- **Ладдер-заголовок секции** (та самая связь план↔детализация):
  - **Лимит** = `plan_cents`
  - **Расписано** = `Σ planned (manual, unposted)`
  - **Свободно в лимите** = `plan_cents − Σ planned_unposted − Σ actual_spent`
  - **Partial OK:** `Расписано < Лимит` → нейтрально/позитивно. `Расписано > Лимит` → **только мягкое предупреждение**, не блокер (потолок, не цель).

### Шаг 3 — «Провести план → факт» на сегодня / свою дату (НОВОЕ)

**3a. Одиночное «Провести».**

- На planned-строке → «Провести» → **выбор даты** (переиспользуем паттерн AddSheet: Сегодня / Вчера / Своя дата).
- Confirm → НОВЫЙ `POST /periods/{id}/planned/{pid}/post` `{tx_date}`:
  - Создаёт `actual_transaction(kind=expense, amount=-abs(amount_cents), category_id, tx_date, source=mini_app, account_id=primary)` через существующий `create_actual_v10` (баланс + roundup).
  - Ставит `planned_transaction.posted_txn_id = actual.id`.
  - Строка → «Проведено · отменить» → НОВЫЙ `unpost` (удаляет actual, чистит линк) — зеркало `unpost_subscription`.
  - Идемпотентность: сервер проверяет `posted_txn_id IS NULL` иначе `409`.

**3b. Bulk «Провести всё на дату»** (хедлайн владельца «одной кнопкой»).

- Кнопка **«Провести запланированное»** сверху детализации/Plan → выбор даты (Сегодня по умолчанию) + список того, что проведётся.
- Confirm → НОВЫЙ `POST /periods/{id}/planned/post-batch` `{planned_ids:[…], tx_date}` → транзакционно прогоняет логику одиночного пост; возвращает созданные actuals + пропущенные (уже проведённые). Тост «ПРОВЕДЕНО · N → реестр».
- **Счёт:** проведение требует `account_id` (как подписки). По умолчанию — primary; пикер счёта только если счетов >1 (см. открытый вопрос).

### Шаг 4 — Видеть связь план↔факт

- CategoryDetail / Home уже показывают План / Факт / В запасе. Расширяем ладдер категории:
  - **План (лимит)** = `plan_cents`
  - **Запланировано** = `Σ planned unposted` (новое)
  - **Факт** = `Σ actual` (есть; теперь включает проведённые planned, т.к. проведение пишет реальный actual)
  - **В запасе** = `plan_cents − Факт` (есть).
- Петля становится видимой: лимит → детализация → проведено в факт → съедает «в запасе».

## 5. Открытые вопросы к владельцу

1. **Жизнь лимита — снапшот per-period или живой шаблон?** Сейчас `Category.plan_cents` — единое живое значение (не per-period, не снапшотится на закрытии). Нужен ли авто-перенос лимита помесячно (шаблон) и/или заморозка лимита per-period при закрытии? Влияет на то, добавляем ли отдельную snapshot-таблицу.

2. **При проведении planned-строки она «съедается» или остаётся?** Подписки остаются (только ставят `posted_txn_id`). Для ручной детализации: проведённая строка **остаётся видимой как «проведено» (можно отменить)** [рекомендуется, обратимо] или **исчезает/конвертируется** в факт целиком? «Съедается» = нельзя отменить/редактировать.

3. **Что именно «настраиваемый день»?** (a) дата на каждое проведение (Сегодня/Вчера/Своя дата, как AddSheet); (b) единая глобальная «дата проведения» (например всегда 1-е); (c) собственный `planned_date` каждой строки. Рекомендация: (a) для одиночного, bulk применяет одну общую дату.

4. **Bulk-post — один actual на planned-строку или один сводный actual на категорию?** Рекомендация: **один actual на строку** (сохраняет описания, обратимость, соответствует детализации). Подтвердить, что слияние в сумму не нужно.

5. **Лимит vs детализация — мягко или жёстко?** Вы сказали partial OK (потолок). Подтвердить: только **предупреждать** при `Σdetail > limit`, никогда не блокировать. И должно ли правило `Σplan ≤ income` (существующее жёсткое) применяться и к детализации, или только к лимитам?

6. **Считается ли детализация в «Факт/в запасе» до проведения?** (a) непроведённая детализация — просто план, влияет на факт только после проведения [рекомендуется, соответствует «post→actual»]; (b) непроведённая детализация заранее уменьшает «в запасе» как прогноз. Выбор меняет математику ладдера в Шаге 4.

7. **С какого счёта финансируется проведённая planned-строка?** Проведение пишет реальный `actual_transaction` с `account_id`. По умолчанию молча primary, или всегда спрашивать? (Подписки требуют счёт или отказывают.)

8. **Доходные категории / доходная детализация?** Income-план сегодня — единое `AppUser.income_cents`, без per-category лимитов. Нужна ли детализация/проведение для **дохода**, или только расход (как AddSheet)?

9. **Отношение к подпискам.** Подписки уже создают `subscription_auto` planned-строки и имеют свой пост. Слить новый «провести запланированное» и существующие «Регулярные · провести» в **одну поверхность**, или оставить раздельно? (Rollup в любом случае должен исключать `subscription_auto`.)

## 6. Liquid Glass — апгрейд

### Почему текущее стекло читается как «херня»

| Сейчас                                               | Проблема                                                                                                       |
| ---------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| `--lgn-card: rgba(255,255,255,0.55)` плоская заливка | Одна равномерная плоскость. Нет направления света, нет кривизны — это «frosted 2018», не Liquid Glass.         |
| `--lgn-glass-border` задан, но **не используется**   | Нет specular-кромки — карточки растворяются в авроре. Edge lensing (#1 признак настоящего стекла) отсутствует. |
| `--lgn-shadow-card: 0 1px 10px /.06`                 | Слишком слабая тень, нет отрыва от фона — карточки «нарисованы», не парят.                                     |
| Аврора 0.38–0.55 alpha                               | Слишком сочно/busy за полупрозрачными карточками → мутит цифры (контраст-риск), дешёвый «обойный» вайб.        |
| `blur(22px) saturate(175%)`                          | Сатурация ок, но при плоской заливке без хайлайта = просто муть.                                               |
| Нет top-light / inner glow / hairline                | Плоско. У реального стекла яркая верхняя кромка, тускнеющая книзу.                                             |

**Ядро фикса:** Liquid Glass = **lensing + specular кромка + слоистая глубина**, а не «убрать прозрачность и блюр». Настоящую рефракцию в CSS не сделать, но «прочтение» убедительно фейкается: слоистые градиенты, направленная верхняя кромка, внутренний glow, правильная elevation.

### Что делает «настоящее стекло»

1. **Edge lensing / specular top-light кромка** — главный признак. Свет сверху: верхняя кромка — яркий hairline (`rgba(255,255,255,.5–.7)`), нижняя — тонкая тёмная. Аппроксимируем градиентной кромкой, не плоским 1px.
2. **Слоистая глубина** — три слоя, никогда glass-on-glass: контент / рефракционный фон / парящее стекло-chrome. Глубина = сильная мягкая тень + яркая верхняя кромка.
3. **Глянцевый translucent с saturation-boost** — `backdrop-filter: blur() saturate(160–180%) brightness(1.04)`. Сатурация отделяет стекло от матового пластика.
4. **Мягкий inner glow (sheen)** — слабый top-down белый градиент _внутри_ карточки на верхней трети → кривая глянцевая поверхность, ловящая свет.
5. **Тонкие hairlines** — сепараторы `0.5px` чётко (не 1px-блюр), низкая alpha.
6. **Сдержанная motion (опц.)** — press-scale 0.97 + один медленный sheen-дрейф на hero-карточке. Непрерывная анимация — нет (стекло должно покоиться). Всё под `prefers-reduced-motion`.
7. **Контраст для финансов (без компромиссов)** — **min 4.5:1** для сумм. План/факт/дельта — на самой непрозрачной зоне карточки или на opaque-тайле, никогда на тонкой прозрачной зоне над авророй.

### Конкретный CSS/токен-план

Все правки — в `frontend/src/stylesV10/native.css` (токены) + `NativePrimitives.module.css` / `NativeShell.module.css` (рецепты). **Скоуп — только тема `liquid_glass`; Maximal Poster не трогаем, pixel-baselines в безопасности** (poster не читает `--lgn-*`).

**Токены (`native.css :root`):**

```css
--lgn-glass-blur: blur(20px) saturate(165%) brightness(1.05);
--lgn-glass-blur-chrome: blur(24px) saturate(180%);

/* Заливка-градиент → глянцевая кривая поверхность, ярче сверху */
--lgn-card: linear-gradient(
  180deg,
  rgba(255, 255, 255, 0.72) 0%,
  rgba(255, 255, 255, 0.52) 38%,
  rgba(255, 255, 255, 0.46) 100%
);
--lgn-card-press: rgba(233, 233, 239, 0.85);

/* Specular-кромки: верх ярче, низ темнее */
--lgn-edge-top: rgba(255, 255, 255, 0.85);
--lgn-edge-bottom: rgba(255, 255, 255, 0.1);
--lgn-edge-hair: rgba(255, 255, 255, 0.55);

/* Inner sheen */
--lgn-sheen: linear-gradient(
  180deg,
  rgba(255, 255, 255, 0.45) 0%,
  rgba(255, 255, 255, 0) 42%
);

/* Elevation: контактная + мягкая ambient */
--lgn-shadow-card:
  0 1px 1px rgba(31, 38, 80, 0.04), 0 6px 20px rgba(31, 38, 80, 0.1);
--lgn-shadow-float:
  0 2px 6px rgba(31, 38, 80, 0.1), 0 18px 48px rgba(31, 38, 80, 0.22);
```

**Glass-card рецепт (для `.insetGroup`, `.circleBtn`, hero):**

```css
.insetGroup {
  position: relative;
  background: var(--lgn-card); /* теперь градиент */
  backdrop-filter: var(--lgn-glass-blur);
  border-radius: var(--lgn-r-card);
  box-shadow:
    var(--lgn-shadow-card),
    inset 0 1px 0 0 var(--lgn-edge-top),
    /* яркая верхняя кромка */ inset 0 -1px 0 0 var(--lgn-edge-bottom); /* тонкая нижняя */
  overflow: hidden;
}
.insetGroup::after {
  /* inner sheen, декоративный */
  content: "";
  position: absolute;
  inset: 0;
  border-radius: inherit;
  background: var(--lgn-sheen);
  pointer-events: none;
  z-index: 0;
}
.insetGroup > * {
  position: relative;
  z-index: 1;
}
```

**Градиентная кромка (самый «lensing» деталь — на hero):** масковый `::before` с `linear-gradient(180deg, var(--lgn-edge-top), …)` + `mask-composite: exclude` для рамки 1px.

**Фон — успокоить, оставить рефракцию:** alpha блобов ~−30%, добавить верхний световой vignette (источник для ярких верхних кромок). Прямо помогает контрасту цифр.

**Chrome (tab/nav):** добавить яркую верхнюю кромку + реальную тень (`inset 0 0.5px 0 rgba(255,255,255,0.6)`, `0 -8px 24px rgba(31,38,80,0.08)`).

**Финанс-readability (в компонентах):** суммы — на нижней (непрозрачной) зоне карточки или на `--lgn-card-solid` тайле; дельта-цвета semantic full-strength (не тонировать стеклом, проверить ≥4.5:1); один акцент на поверхность (бренд-оранж только для primary-action); никакого glass-on-glass.

**Fallback (`@supports not backdrop-filter`):** расширить — переопределить `--lgn-card` на opaque, обнулить `--lgn-sheen` и `--lgn-edge-*`, иначе sheen сломается на непрозрачных карточках.

### Порядок раскатки (риск ↑ payoff)

1. Тени + specular-кромки (`--lgn-shadow-card`, inset top/bottom) — макс. реализм, 0 нового DOM. ~30 мин.
2. Градиентная заливка + inner sheen `::after` (нужна z-index-обёртка детей). ~1ч.
3. Успокоить + подсветить фон. ~20 мин.
4. Material tweak (`saturate 165% brightness 1.05`). Тривиально.
5. Градиентная кромка hero (масковый `::before`) — «wow», сначала только на дашборд-hero.
6. Motion + усиление fallback. ~30 мин.

После шагов 1–3 уже перестанет читаться как «херня».

### Guardrails

- Все правки гейтятся `ui.theme === 'liquid_glass'` через `--lgn-*` → poster pixel-baselines не двигаются.
- У `liquid_glass`-шелла **нет коммитнутого Linux pixel-baseline** сегодня (baselines — poster/v06). Подтвердить перед добавлением, иначе визуальные правки потребуют рефреша baseline на Linux.
- `backdrop-filter` + стопка градиентов на скролле дороги по paint на слабых устройствах/TMA — тяжёлый рецепт только на карточки/chrome, не на каждую `.insetRow`; `will-change` не вешать на длинные списки.

### Что прислать референсами (нужны реальные скриншоты)

Текст-фетч не увидел мокапы QuickBooks/liquidglassdesign (JS-галерея) и HIG Materials (JS-render). Чтобы клонировать, а не аппроксимировать, нужны 2–3 скриншота, чтобы определить:

- **Тинт стекла** — нейтрально-белый frosted или цветной (cool blue / warm)? Меняет `--lgn-card` и ощущение сатурации.
- **Фон** — спокойный near-solid градиент vs яркая аврора vs фото/mesh. Наша аврора — догадка.
- **Интенсивность кромки** — subtle Apple-style vs тяжёлый глянцевый «iOS clear».
- **Плотность карточек и радиус** — возможно крупнее (20px) и больше паддинга, чем наш 14px iOS-grouped.
- **Подача data-viz** — суммы на opaque-тайлах, цветных чипах или прямо на стекле?
- **Акцент** — оставить бренд-оранж или матчить акцент референса?
