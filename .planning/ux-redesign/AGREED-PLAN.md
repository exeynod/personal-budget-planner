# Согласованный план: UX-реворк планирования бюджета + Liquid Glass v2

Статус: **решения владельца зафиксированы** (две сессии вопросов). Ниже — финальная
согласуемая модель. Реализация начинается ТОЛЬКО после финального «ок» владельца.
Источник ресёрча: `RESEARCH.md`. Визуал стекла: `lg-visual-analysis.md` + `lg-refs/`.

## A. Решения владельца (зафиксировано)

1. **Лимит** — через **шаблон плана** + авто-перенос помесячно.
2. **Шаблон vs план месяца — ДВЕ РАЗНЫЕ СУЩНОСТИ:**
   - **Шаблон плана** — задаётся один раз, иногда правится. Несёт **лимиты + детализацию**
     (включая повторяющиеся строки) по категориям. Авто-применяется к новому периоду.
   - **План на месяц** — инстанс шаблона на конкретный период. Правится локально для
     месяца; в шаблон НЕ возвращается; на следующий месяц НЕ влияет (там снова чистое
     применение шаблона).
3. **Проведение** — у каждой запланированной строки своя `planned_date`; кнопка проводит
   строку в факт на эту дату.
4. **После проведения** — строка остаётся как «проведено» (обратимо, есть «отменить»).
5. **Непроведённая детализация** — план-онли: «Факт»/«в запасе» меняются только после
   проведения (чистое post→actual).
6. **Доходы** — планирование/детализация/проведение нужны и для доходов (не только расход).
7. **Подписки** — слить в одну поверхность планирования (подписки = вид повторяющейся
   детализации; одна кнопка «провести»).
8. **Лимит = потолок** — превышение детализацией = мягкое предупреждение, НЕ блок.
9. **Bulk-проведение** — один факт на строку (сохраняет описания/обратимость).
10. **Счёт проведения** — primary по умолчанию; пикер только если счетов >1.
11. **Liquid Glass фон** — «оптический» цветной (как Payrix ref1) + апгрейд стекла.

## B. Целевая data-модель

**Шаблон (reusable, НЕ per-period):**

- `plan_template_item` (ОЖИВИТЬ — был дропнут в alembic 0013): `user_id, category_id,
limit_cents` — лимит категории в шаблоне.
- `plan_template_line` (НОВОЕ): повторяющиеся строки детализации шаблона:
  `user_id, category_id, title, amount_cents, day_of_period (опц.), kind(expense|income)`.
- Подписки (`subscription`) — это уже повторяющиеся строки; в шаблон-поверхности они
  показываются как template-lines с источником `subscription` (не дублируем сущность,
  переиспользуем существующую + reminders/auto-charge остаются).

**План на месяц (per-period instance):**

- Per-period лимит категории: новая таблица `period_category_plan
(period_id, category_id, limit_cents)` ИЛИ снапшот в `budget_period`. (Category.plan_cents
  глобально-живой больше не источник истины для лимита месяца.)
- `planned_transaction` (СУЩЕСТВУЕТ, per-period): строки детализации месяца —
  `period_id, category_id, title/description, amount_cents, planned_date, kind,
source(manual|subscription_auto), posted_txn_id (НОВОЕ FK→actual_transaction)`.

**Мост план↔факт:**

- `planned_transaction.posted_txn_id FK → actual_transaction (ON DELETE SET NULL)` (НОВОЕ).

**Apply-template:** при создании периода (close_period job / первый вход) — копировать
шаблон (лимиты `plan_template_item` → period_category_plan; строки `plan_template_line`

- активные подписки → planned_transaction с `planned_date` из day_of_period). Идемпотентно.

## C. UX-сценарий (web native, потом iOS)

### Экран «Шаблон плана» (Управление → Шаблон бюджета)

- Список категорий; на каждой: **лимит** (инлайн-поле ₽) + раскрытие **детализации шаблона**
  (повторяющиеся строки: title, сумма, день периода). CRUD строк.
- Расход/Доход — сегмент (доходные категории тоже).
- Это «задаю один раз и иногда правлю».

### Экран «План месяца» (Home → «План месяца»)

- Инстанс шаблона на текущий период. Сверху — «Осталось распределить» (как сейчас).
- На каждой категории: **лимит** (правится локально на месяц) + раскрытие
  **«Детализация лимита»**:
  - Ладдер: **Лимит / Расписано (Σ planned unposted) / Свободно**. Превышение → мягкое
    предупреждение.
  - Список planned-строк (manual + подписки в одной поверхности): title, сумма,
    `planned_date`, статус (запланировано / проведено), кнопка **«Провести»** / «Отмена».
  - «+ Добавить запланированную трату» (title, сумма, дата) → `POST /planned`.
- Кнопка **«Провести запланированное»** (bulk) — проводит выбранные строки, каждую на её
  `planned_date` (или общий выбор даты для пакета — уточнить в реализации), один факт на
  строку. Тост «Проведено N → реестр».

### Связь план↔факт (Home / CategoryDetail)

- Ладдер: **Лимит / Запланировано (unposted) / Факт (вкл. проведённые) / В запасе**.
- Видимая петля: лимит → детализация → провёл → факт растёт, «в запасе» падает.

## D. Liquid Glass v2 (оптический)

- Фон: насыщенный «оптический» цветной glass-mesh (ref1 Payrix) — притушить под контентом
  для читаемости цифр (≥4.5:1).
- Карточки: specular верхняя кромка + inner sheen + слоистые тени + градиентная заливка
  (см. рецепты в RESEARCH.md §6), крупнее радиус.
- Floating glass-таб-бар (пилюля) со specular-кантом (ref4 iOS 26).
- Круглые glass-кнопки.
- Дата-виз план/факт: кольца/сегментные бары/вертикальный гейдж лимита (ref2/ref3).
- Скоуп — только `ui.theme='liquid_glass'`; Maximal Poster не трогаем (pixel-baselines).

## F. Открытые микро-решения (беру дефолты, скажите если не так)

- Точка входа шаблона: Управление → «Шаблон бюджета» (строка уже есть в хабе).
- Apply-template: авто при создании периода + кнопка «применить шаблон заново» в Плане месяца.
- Bulk-дата: по умолчанию каждая строка на свою `planned_date`; опц. «провести всё на одну дату».
- Доходные категории: вводятся как kind=income категории (нужен CRUD категорий по kind).

## G. Выпиливание лишнего функционала (решение владельца)

Убрать (backend + оба web-шелла + iOS + тесты + навигация + онбординг):

1. **Накопления (Savings/Копилка)** — полностью: экран Savings, цели (goals), округление
   (roundup config/base), депозиты, виды операций `roundup`/`deposit`, системная категория
   `savings`, строка «Копилка» в хабе, любые ссылки/фильтры savings. Onboarding — убрать
   savings-сиды.
2. **Управление счетами** — убрать экран «Счета», деталь счёта, пикер счёта в AddSheet,
   «Перевод», строку «Счета» в хабе. **ОСТАВИТЬ один неявный баланс:** под капотом один
   primary-`account` на юзера; `actual_transaction.account_id` остаётся = этот primary;
   Home «Остаток» = стартовый + доход − расход (+ корректировка, см. H). Онбординг
   продолжает спрашивать стартовый баланс (но без понятия «счета»).
3. **Пауза категории** — убрать поле/тумблер `paused` и UI «Пауза/Включить» (Plan,
   CategoryDetail). (Удаление/`is_archived` категорий — НЕ трогаем, это другое.)
4. **Rollover категории** («Остаток → Прочее/Накопления») — убрать поле `rollover` и чипы
   (Plan/CategoryDetail). С уходом накоплений оно бессмысленно.

## H. Корректировка остатка (НОВАЯ фича — запрос владельца)

Реальный остаток иногда расходится с расчётным. Нужна **ручная сверка**:

- В Настройках — действие **«Привести остаток»**: ввожу свой реальный остаток.
- Приложение создаёт **балансирующую корректировку** = `(реальный − текущий расчётный)`,
  чтобы «Остаток» стал равен введённому, независимо от транзакций. Реализация: отдельная
  запись-корректировка (тип `adjustment`) или поле `balance_adjustment_cents` у юзера,
  прибавляемое к расчёту. Предпочтительно — явная запись «Корректировка остатка» в реестре
  (обратимо, видно в истории), либо накопительный offset. Уточнить в реализации; на UX:
  одно поле + «Применить».
