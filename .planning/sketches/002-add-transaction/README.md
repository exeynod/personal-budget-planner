# Sketch 002: Add Transaction

## Design Question
UC-2: добавление факт-транзакции через Mini App. Должно быть быстрым (это
самое частое действие в продукте). Какой паттерн лучше: полноэкранный с
кастомной клавиатурой, выезжающий bottom-sheet поверх дашборда, или wizard
из трёх шагов?

## How to View
```
open .planning/sketches/002-add-transaction/index.html
```

## Variants
- **A: Fullscreen + numeric keypad** — Полноэкранная форма, большая сумма сверху, под ней пилюля категории + поля «Дата/Описание», ниже — кастомная цифровая клавиатура. Банковский стиль (Tinkoff/Revolut). Минимум tap'ов, всё на одном экране.
- **B: Bottom sheet** — Выезжающая снизу панель поверх затемнённого дашборда. Контекст (баланс, категории) виден за shadow. Стандартный системный input для суммы, поля-карточки для остальных. Хорошо для «быстро записал и закрыл».
- **C: Wizard 3 шага** — Прогресс-бар + три экрана: сумма (с быстрыми кнопками 100/500/1k/5k), сетка категорий 2×3 с показом текущего использования бюджета, детали + summary-блок. Дружелюбно для нового пользователя, медленнее для опытного.

## What to Look For
- **Скорость**: сколько tap'ов от «открыл форму» до «сохранил». A — 1 экран, B — 1 sheet, C — 3 шага.
- **Контекст**: видно ли в момент ввода, сколько уже потрачено по этой категории? B — нет, A — нет, C — да (показывает «12 500 / 50 000» в tile).
- **Клавиатура**: кастомный keypad (A, C) даёт визуальный контроль, но дублирует системную; нативная (B) быстрее открывается, но скрывает половину экрана.
- **Ошибки**: где удобнее править введённое (вернуться, изменить категорию).
- **TG-нативность**: bottom sheet (B) — самый «телеграмный», keypad (A) — самый «банковский», wizard (C) — нейтральный.
- **Первое впечатление vs daily-use**: какой выиграет в долгой эксплуатации.
