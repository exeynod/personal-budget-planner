# Sketch 006: Onboarding

## Design Question
UC-10: первый запуск Mini App. Нужно: привязать chat (для push), ввести
стартовый баланс, выбрать `cycle_start_day`, опционально засеять категории.
Какая раскладка лучше: классический wizard, всё-в-одном-скролле или гибрид
с welcome-экраном?

## How to View
```
open .planning/sketches/006-onboarding/index.html
```

## Variants
- **A: Linear wizard (4 шага)** — Welcome → bot bind → balance+cycle → seed категорий. Каждый шаг = отдельный экран с прогресс-баром. Показан шаг 2 (bot bind) — приглашение открыть бота с иллюстрацией. Шаги переключаются MainButton.
- **B: Single scrollable page** — Все шаги в одном скролле. Каждая секция нумерована (1/2/3/4), завершённые получают зелёную галку, активные — синюю. Прогресс ощутим, но без принудительной линейности. Показан состояние «1 готов, 2-3-4 заполняются».
- **C: Welcome → 2-step wizard** — Полноценный welcome-экран (gradient icon, фичи списком, кнопки «Начать» и «Пропустить»). Дальше короткий wizard всего из 2 шагов: bot bind + (balance+cycle+seed на одном экране). Минимум переходов.

## What to Look For
- **Скорость**: A — 4 экрана, B — 1 длинный скролл, C — welcome + 2 wizard.
- **Контроль**: A — самый дисциплинированный, нельзя пропустить шаг; B — можно прыгать; C — компромисс.
- **Психология первого впечатления**: C даёт «маркетинговый» welcome (фичи списком, gradient icon) — выглядит более premium, но добавляет один тап.
- **Bot bind как блокер**: во всех трёх вариантах это критичный шаг (без него push не работает). В A — отдельный шаг, в B — секция №1, в C — шаг 1 wizard.
- **Skip-сценарий**: только в C есть «Пропустить». Это уместно для skip-frequent юзеров, но требует логики «частичного» state'а.
- **Cycle day stepper**: показан только в B. Лучше «−/число/+» или dropdown 1..28? (deferred design).

## Notes
- Гибрид C логично объединяет 3 шага в 1 экран только если они умещаются в 375px без скролла. На моке еле умещаются.
- B даёт ощущение «всё видно, я в контроле», но требует scroll-анализа от пользователя.
- A — наиболее «системный» паттерн TG/iOS.
