## 0. Контекст и идея

**TG Budget** — приложение личного бюджета внутри Telegram (Mini App). Помогает
пользователю **планировать**, **записывать**, **анализировать** траты и **копить**
по целям. Не банк — учётная система поверх ручного ввода и (опционально) импорта
из банков.

**Tone of voice / визуал:** «Maximal Poster» — печатная типографика, большие цифры
JetBrains Mono, акценты Italic-сериф, плотные плашки, минимум градиентов, никаких
«мягких карточек» и эмодзи. Палитра — кораллово-кобальтовая со вспышками жёлтого.

**Платформа:**

- **MVP:** Telegram Mini App (TWA), web-only, mobile-first 390×844.
- **Backend:** REST/GraphQL, JWT через Telegram InitData.
- **Sync:** опционально импорт операций из банков (Т-Банк, Сбер) — не в MVP.

## 1. Ключевые продуктовые принципы

1. **Один тап — одна запись.** FAB всегда виден, форма — bottom-sheet с цифровой клавой.
2. **План всегда на виду.** Главный экран = «дневной темп» × «осталось дней» — числа,
   а не графики.
3. **Остаток категории не теряется.** Если в конце месяца по категории остался лимит,
   он переезжает либо в **«Прочее»**, либо в **«Накопления»** — выбор пользователя на
   уровне категории.
4. **AI как помощник, не как чат-бот.** Открывается на яркое наблюдение по реальным
   данным («Май в плюсе на 21 170 ₽…»), а не на пустое поле.
5. **Копилка — нативная фича.** Округление трат → копилка, цели с прогрессом, депозиты
   видны в реестре операций.
6. **Никакой «милоты».** Нет иконок-эмодзи, нет «весёлых» иллюстраций. Цвет, тип, цифра.

## 2. Архитектура навигации

```
┌─────────────────────────────────────────┐
│  ОНБОРДИНГ (4 шага, при первом запуске) │
│  → ставит initial state                 │
└─────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│  TAB BAR · 5 кнопок (с FAB по центру)   │
│  ▣ Главная   ◊ Копилка   +   ✦ AI   ⌘ Управление │
└─────────────────────────────────────────┘
```

**Стек экранов** (push/pop, slide-анимация):

- **Главная** → Категория (детали)
- **Главная** → Все операции (Реестр)
- **Главная** → PLAN мая (бюджет на месяц)
- **Главная** → Счета → Счёт (история)
- **Управление** → Аналитика
- **Управление** → PLAN мая → Подписки
- **Копилка** (таб)
- **AI** (таб)
- **Add Sheet** (bottom-sheet поверх любого экрана, FAB)

**Анимации переходов:**

- `fwd` (push) — slide-in справа (28px, 420ms, ease-out cubic).
- `back` (pop) — slide-in слева.
- `tab` (переключение таба) — fade+rise (350ms).

## 3. Бизнес-правила

### 3.1 Месяц и период

- Месяц = календарный (1–28/30/31 число).
- Все цифры на «Главной» и в «PLAN» считаются от 1-го числа текущего месяца.
- В аналитике диапазоны: текущий месяц, предыдущие два, ниже — «полгода / год».

### 3.2 План (PLAN мая)

**Сущности:** доход (`income`), категории (`Category[]` с `plan` и `rollover`),
регулярные платежи (`Recurrent[]` — аренда, ЖКУ, подписки).

**Правила:**

- Сумма всех `plan` ≤ `income`. Если больше — статус «OVER», блок CTA «продолжить»
  до фикса.
- `income - sum(plan)` = «Осталось распределить». Может быть положительным (=
  buffer / накопления по умолчанию).
- В PLAN видно текущий **факт** на ползунке: ползунок = лимит, поверх — узкая
  заливка факта (жёлтая, или красная если `act > plan`).
- Лимит редактируется двумя способами: ползунок (шаг 500 ₽) **и** тап по числу →
  ввод вручную.
- Регулярные платежи отображаются отдельным блоком «Регулярные · провести в
  факт» — каждый можно вручную отметить «проведено» (попадёт в реестр со
  спецтегом).

### 3.3 Перенос остатка категории (rollover)

- На уровне **категории** настраивается, куда уйдёт неиспользованный лимит в конце
  месяца: `misc` (прочее) или `savings` (накопления).
- На экране PLAN сводный блок «Остаток по итогу месяца» суммирует все остатки
  по двум направлениям.
- На детальном экране категории показывается, куда «уйдёт» остаток (плашка вверху).

### 3.4 Копилка

**Источники пополнения:**

1. Округление трат: каждая трата округляется вверх до 10 / 50 / 100 ₽ (выбирается
   пользователем); разница уходит в копилку.
2. Перенос остатка категории (см. 3.3).
3. Ручной депозит (кнопка «Пополнить» на экране Копилки).

**Цели:**

- Каждая цель — `{ name, target, cur, due }`. `due` опционален.
- Прогресс = `cur / target` (округление до целого %).
- Из копилки можно «выдать» под цель (не в MVP).

**Транзакции:**

- Каждое пополнение копилки попадает в реестр как операция категории «накопления»
  с тегом `roundup` или `deposit` (визуально выделяется жёлтой плашкой).

### 3.5 Транзакция

**Сущность:**

```ts
type Txn = {
  id: string;
  date: ISODate;       // Sun 2026-05-09T14:32:00+03:00
  amount: number;      // в копейках, отрицательное = расход, положительное = доход
  name: string;        // "Surf Coffee"
  catId: string;       // "cafe"
  accountId: string;   // "tink"
  kind?: 'income' | 'expense' | 'roundup' | 'deposit';
};
```

**Правила:**

- Расходы пишутся отрицательными.
- Округление и депозиты в копилку — отдельный `kind`, фильтруются как «копилка» в
  реестре.

### 3.6 Счета

- Несколько счетов, у одного `primary: true` (по умолчанию для новых трат).
- Типы: `card`, `cash`, `savings` (накопит. счёт).
- Сумма всех счетов = «в кошельке» на главной.
- Тап по счёту → история операций по этому счёту.

## 4. Онбординг (4 шага)

Запускается один раз при первом старте. Заполняет initial state.

| #  | Экран                                | Что собираем                                                         | Валидация                |
|----|--------------------------------------|----------------------------------------------------------------------|--------------------------|
| 01 | **Доход**                            | `income` (число, ≥0)                                                 | `income > 0` для NEXT    |
| 02 | **Счета**                            | `accounts[]` (мин. 1, можно несколько)                               | Не пустой массив         |
| 03 | **План**                             | `plan: Record<catId, number>` — слайдеры по 8 дефолтным категориям   | `sum(plan) ≤ income`     |
| 04 | **Цель** (опц.)                      | `goal?: { name, amount }`                                            | Можно пропустить         |

После 04 — экран «ВСЁ. деньги — под контролем.» с резюме (доход, счета, план,
цель) и большой кнопкой «НАЧАТЬ →» → переход на Главную.

**UX-замечания:**

- На каждом шаге сверху прогресс-полоска (4 деления).
- Кнопка «← НАЗАД» работает в т.ч. с шага 01 — закрывает онбординг (или вернёт
  на сплэш, если он будет).
- На шаге 03 «Распредели X ₽» в реальном времени пересчитывается «остаётся X ₽
  → накопления» внизу. Если ушли в минус — красный бейдж и блок NEXT.

## 5. Главная

См. `SCREENS.md → 01.HOME`. Кратко:

- **Eyebrow:** `VOL.04 / MAY 2026 · 23 ДНЯ` — номер месяца, год, осталось дней.
- **Hero:** «Дневной темп — **3 200 ₽**» — главное число, count-up при входе.
- **Подложка:** «осталось 23 дня · в кошельке 142 380 ₽ →» (тап → счета).
- **Plan-bar:** строка «PLAN МАЯ · + 21 170 ₽ →» (тап → PLAN).
- **Категории:** список (отсортирован по `act/plan`, превышения сверху). У каждой
  — название, % использования, плашка `OVER` если превышение, прогресс-бар, факт/план.
- **Цвет:** настраивается (Tweak: коралл / кобальт / крем). Mobile dark по
  умолчанию = коралл.

**Пустые состояния:**

- Нет категорий → CTA «настроить план» (направляет в PLAN мая).
- Нет операций → «дневной темп» = `plan / daysLeft`, но категории показывают 0%.

## 6. AI Assistant

См. `SCREENS.md → 03.AI`.

**Принцип:** не «чат-бот с пустым полем», а «карточка с наблюдением + 4 подсказки
+ инпут».

**Стартовое наблюдение:** генерируется бэкендом на основе реальных данных. Шаблоны:

- «Май в плюсе на X ₽. / Кафе уже +N% к лимиту — стоит притормозить.»
- «За неделю экономия Y ₽ — на 12% лучше прошлой.»
- «Завтра списание подписок на Z ₽.»

**Подсказки (chips):** 4 шт., обновляются по контексту:

- «Сколько я потратил на еду?»
- «Запиши: кофе 350 ₽»
- «На что трачу больше всего?»
- «Шаблон на отпуск»

**Действия AI** (после уточнения с пользователем):

- Записать трату из текста («Запиши кофе 350 ₽»).
- Предложить шаблон бюджета.
- Пересчитать план (сценарий «отпуск +130 000 в июне»).
- Вернуть факт по периоду / категории.

**Формат ответа AI:**

- Сериф-италик, размер 17px, без аватарки.
- Реплики пользователя — чёрная плашка справа.
- Индикатор «печатает» — три точки с анимацией.

**Приватность:** запросы уходят на бэкенд, не на сторонние LLM напрямую (через
proxy). История хранится локально (опционально шифруется).

## 7. Реестр операций

См. `SCREENS.md → 02.TXN`.

- Сгруппирован по дням («Сегодня», «Вчера», «7 мая» — DM Serif italic 28px).
- Каждый день: сумма за день справа, ниже список операций.
- Каждая операция: время · название · категория · счёт · сумма (моно).
- **Чипы-фильтры** сверху: Все / Кафе / Продукты / Транспорт / Подписки / Копилка.
- Транзакции копилки помечены жёлтой плашкой «↻ ОКРУГЛ.» или «→ КОПИЛКА».

**MVP-функциональность:**

- ✅ Просмотр + фильтрация.
- ✅ Тап по операции → редактирование (R5).
- ✅ Свайп-влево по операции → удалить с подтверждением (R5).

## 12. Add Sheet (новая операция)

См. `SCREENS.md → 10.ADD`.

- **Sheet** снизу, чёрный фон.
- **Сумма:** жёлтым 86px, ввод цифровой клавой 3×4.
- **Описание:** italic-сериф, плейсхолдер «кафе / продукты / …».
- **Когда:** Сегодня / Вчера / Своя дата.
- **Категория:** chips, выбор обязателен.
- **Счёт:** строка с текущим основным, тап → выбор.
- **Сохранение:** жёлтый CTA, активен только при сумме > 0 и выбранной категории.
- **Закрытие при изменениях:** confirm «Сумма X ₽ не сохранена. Продолжить /
  Отменить».

## 14. Acceptance criteria для MVP

1. **Онбординг** проходится за < 60 секунд, валидирует ввод, ставит корректный initial state.
2. **Главная** показывает дневной темп, кошелёк, plan-бейдж, отсортированные категории.
3. **Add Sheet** записывает трату за один тап (не считая ввода суммы), закрывается с
   подтверждением при потере данных.
4. **PLAN** позволяет менять лимиты, видеть остаток к распределению, помечать
   регулярные платежи проведёнными.
5. **AI** даёт релевантное наблюдение, реагирует на 4 пресет-подсказки, принимает
   свободный ввод.
6. **Копилка** показывает накопления, цели с прогрессом, переключатель округления.
7. **Все экраны** работают в тёмной палитре приложения, шрифты прогружены, нет
   FOUT-моментов.
