Migration 0007 idempotent pattern (DO-block):
```sql
DO $$ BEGIN
  IF NOT EXISTS (SELECT 1 FROM pg_roles WHERE rolname = 'budget_app') THEN
    CREATE ROLE budget_app NOLOGIN NOSUPERUSER NOBYPASSRLS;
  END IF;
END $$;
-- Set/update password and login (always runs — accommodates rotation).
ALTER ROLE budget_app WITH LOGIN PASSWORD :pwd;
-- Grants on existing objects.
GRANT USAGE ON SCHEMA public TO budget_app;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO budget_app;
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO budget_app;
-- Default privileges for future objects (alembic creating tables under
-- budget user → grant to budget_app automatically).
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO budget_app;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT USAGE, SELECT ON SEQUENCES TO budget_app;
```

NOTE: ALTER ROLE password parameterization in plain SQL — Postgres ALTER ROLE
syntax не поддерживает bind params. Alembic execute через `op.execute(sa.text(...).bindparams(...))`
ТАКЖЕ не работает с DDL (PostgreSQL parser rejects). → Solution: build SQL string with
quote_literal — escape password through double-single-quote replace или
использовать `format_quote_literal` через PostgreSQL function.

→ Pragmatic safe approach: use SQLAlchemy `text(f"ALTER ROLE budget_app WITH LOGIN PASSWORD {sa.literal(pwd).compile(compile_kwargs={'literal_binds': True})}")` — too messy.

→ Lightweight: Postgres has built-in `quote_literal()` function. Use:
```python
op.execute(
    sa.text("SELECT format('ALTER ROLE budget_app WITH LOGIN PASSWORD %L', :pwd)")
    .bindparams(pwd=password)
)
# Then execute returned string. But execute returns a row, not executes DDL...
```

→ Cleanest: escape password ourselves and embed in raw string:
```python
def _escape_pg_password(pwd: str) -> str:
    """Escape single quotes по PostgreSQL string literal rules."""
    return "'" + pwd.replace("'", "''") + "'"

password = os.environ['BUDGET_APP_PASSWORD']
if not password or password.startswith("change_me"):
    raise RuntimeError("BUDGET_APP_PASSWORD not set or placeholder")
escaped = _escape_pg_password(password)
op.execute(f"ALTER ROLE budget_app WITH LOGIN PASSWORD {escaped}")
```

Это решает inject через single-quote escape (двойная квота — стандартный Postgres lit). Документировать в migration docstring.

    ```python
    """postgres role split: create budget_app NOSUPERUSER NOBYPASSRLS

    Revision ID: 0007_postgres_role_split
    Revises: 0006_multitenancy
    Create Date: 2026-05-07

    Phase 12 D-11-07-02: split single Postgres role into two roles.

    Background: until this migration, runtime processes (api/bot/worker)
    connected as `budget` which is SUPERUSER (created by Postgres entrypoint
    via POSTGRES_USER). PostgreSQL bypasses RLS for superusers, so all the
    Phase 11 RLS policies (alembic 0006) did not actually enforce at runtime
    — only in tests via SET LOCAL ROLE budget_rls_test workaround.

    What this revision does:
      1. Idempotently CREATE ROLE budget_app NOLOGIN NOSUPERUSER NOBYPASSRLS
         (DO-block).
      2. ALTER ROLE budget_app WITH LOGIN PASSWORD :pwd (read from
         BUDGET_APP_PASSWORD env; fail loud if missing/placeholder).
      3. GRANT USAGE ON SCHEMA public + SELECT/INSERT/UPDATE/DELETE
         on ALL TABLES + USAGE/SELECT on ALL SEQUENCES.
      4. ALTER DEFAULT PRIVILEGES so future tables created by `budget`
         (e.g. via subsequent alembic revisions) auto-grant to budget_app.

    Operational notes:
      - This migration must run as the privileged role (DATABASE_URL pointing
        at `budget`). entrypoint.sh sets ADMIN_DATABASE_URL accordingly.
      - After upgrade, runtime DATABASE_URL (used by api/bot/worker) must
        switch to `budget_app:$BUDGET_APP_PASSWORD@...`. docker-compose
        provides both URLs to each service env (Plan 12-05 docker changes).
      - BUDGET_APP_PASSWORD env var must be set; missing → RuntimeError.
        Generate with `python -c "import secrets; print(secrets.token_urlsafe(32))"`.

    downgrade:
      Drops the role + owned objects. Existing data tables remain (owned by
      `budget`), only privileges are revoked.
    """
    from __future__ import annotations

    import os
    from typing import Sequence, Union

    from alembic import op


    revision: str = "0007_postgres_role_split"
    down_revision: Union[str, None] = "0006_multitenancy"
    branch_labels: Union[str, Sequence[str], None] = None
    depends_on: Union[str, Sequence[str], None] = None


    _PLACEHOLDER_PREFIXES = ("change_me", "changeme", "")


    def _resolve_app_password() -> str:
        """Read BUDGET_APP_PASSWORD env var; fail loud if missing or placeholder."""
        pwd = os.environ.get("BUDGET_APP_PASSWORD", "")
        if not pwd or pwd.startswith(_PLACEHOLDER_PREFIXES):
            raise RuntimeError(
                "0007_postgres_role_split: BUDGET_APP_PASSWORD env var must be set "
                "to a real (non-placeholder) value. Generate with: "
                "python -c \"import secrets; print(secrets.token_urlsafe(32))\""
            )
        return pwd


    def _escape_pg_string_literal(s: str) -> str:
        """Escape value as single-quoted Postgres string literal.

        Postgres standard escape: double single quotes inside the literal.
        This is sufficient for ALTER ROLE PASSWORD which does not interpret
        backslash escapes (no E'' prefix needed).
        """
        return "'" + s.replace("'", "''") + "'"


    def upgrade() -> None:
        password = _resolve_app_password()
        password_lit = _escape_pg_string_literal(password)

        # Step 1: idempotent CREATE ROLE budget_app NOLOGIN.
        # NOLOGIN сначала — переключим на LOGIN после установки password
        # чтобы не оставлять role без password в окне между двумя statements.
        op.execute(
            "DO $$ BEGIN "
            "IF NOT EXISTS (SELECT 1 FROM pg_roles WHERE rolname = 'budget_app') "
            "THEN CREATE ROLE budget_app NOLOGIN NOSUPERUSER NOBYPASSRLS; "
            "END IF; "
            "END $$;"
        )

        # Step 2: set password + enable LOGIN.
        # ALTER ROLE password не принимает bind params — escape literal manually.
        op.execute(
            f"ALTER ROLE budget_app WITH LOGIN NOSUPERUSER NOBYPASSRLS "
            f"PASSWORD {password_lit}"
        )

        # Step 3: GRANTs on existing objects.
        op.execute("GRANT USAGE ON SCHEMA public TO budget_app")
        op.execute(
            "GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public "
            "TO budget_app"
        )
        op.execute(
            "GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO budget_app"
        )

        # Step 4: default privileges для future tables (created by current_user
        # which is the privileged role running this migration).
        op.execute(
            "ALTER DEFAULT PRIVILEGES IN SCHEMA public "
            "GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO budget_app"
        )
        op.execute(
            "ALTER DEFAULT PRIVILEGES IN SCHEMA public "
            "GRANT USAGE, SELECT ON SEQUENCES TO budget_app"
        )

        # Step 5: pgvector extension (Phase 10) needs USAGE for budget_app
        # to use vector ops. EXTENSION objects use schema-level permissions —
        # already granted via USAGE ON SCHEMA. No-op here, documented for clarity.


    def downgrade() -> None:
        # Reverse default privileges (best-effort — IF EXISTS не работает на
        # ALTER DEFAULT PRIVILEGES, но REVOKE без-existence guard not fatal).
        try:
            op.execute(
                "ALTER DEFAULT PRIVILEGES IN SCHEMA public "
                "REVOKE SELECT, INSERT, UPDATE, DELETE ON TABLES FROM budget_app"
            )
            op.execute(
                "ALTER DEFAULT PRIVILEGES IN SCHEMA public "
                "REVOKE USAGE, SELECT ON SEQUENCES FROM budget_app"
            )
        except Exception:
            # Role may already be dropped — that's fine for downgrade idempotency.
            pass

        # DROP OWNED BY освобождает все privileges role'у.
        op.execute(
            "DO $$ BEGIN "
            "IF EXISTS (SELECT 1 FROM pg_roles WHERE rolname = 'budget_app') "
            "THEN DROP OWNED BY budget_app; "
            "END IF; "
            "END $$;"
        )
        # Затем DROP ROLE.
        op.execute(
            "DO $$ BEGIN "
            "IF EXISTS (SELECT 1 FROM pg_roles WHERE rolname = 'budget_app') "
            "THEN DROP ROLE budget_app; "
            "END IF; "
            "END $$;"
        )
    ```

| T-12-05-01 | E (Elevation) | Runtime app process inadvertently uses budget (super) URL | mitigate | docker-compose passes budget_app DSN as DATABASE_URL; ADMIN_DATABASE_URL is separate var, only entrypoint.sh exports it for alembic step |
| T-12-05-02 | T (Tampering) | Migration ordering bug — grants applied before policies | accept | RLS policies created in 0006 (Phase 11) BEFORE 0007 grants → policies ready when budget_app first connects (logical order intact). Reverse-order issue impossible because 0007 depends on 0006 |
| T-12-05-03 | I (Info Disclosure) | BUDGET_APP_PASSWORD logged in alembic stdout | mitigate | Migration uses op.execute with literal-escaped password; alembic logs DDL but password in plain literal — accept (operator-trusted env). Documented in migration docstring |
| T-12-05-04 | E | budget_app granted excess privileges (e.g., CREATE/DROP) | mitigate | Migration grants ONLY SELECT/INSERT/UPDATE/DELETE on tables + USAGE/SELECT on sequences; NO CREATE / NO ALTER / NO trigger privileges. Verified via grep on migration source |
