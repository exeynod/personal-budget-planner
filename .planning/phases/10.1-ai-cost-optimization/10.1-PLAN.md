# Phase 10.1 Plan: AI Cost Optimization

11 fixes grouped into 4 implementation waves. Single-developer flow,
~30 minutes work + verification.

## Wave 1 — Token reduction (highest impact)

### T-1.1: Translate `SYSTEM_PROMPT` to English
- `app/ai/system_prompt.py:12-19`
- Drop the «cache_control» comment lie (note in next task)
- Add explicit "Reply in Russian" instruction at the end
- Acceptance: prompt ≤ 70 tokens vs current ~80 (Cyrillic) → 2× token compaction

### T-1.2: Translate `TOOLS_SCHEMA` descriptions and parameter docs to English
- `app/ai/tools.py:260-321`
- All 4 tools' `description` fields + parameter `description` fields
- Tool `name` stays the same (canonical identifier)
- Acceptance: schema serialized = ~80 tokens vs current ~200

### T-1.3: Reduce `AI_MAX_CONTEXT_MESSAGES` 20 → 8
- `app/core/settings.py:39`
- One-line change

### T-1.4: Remove decorative `cache_control` block
- `app/ai/system_prompt.py:34-47`
- Replace structured content with plain string
- Add comment explaining why caching doesn't apply at this scale

## Wave 2 — Observability

### T-2.1: Capture `usage` from OpenAI stream
- `app/ai/providers/openai_provider.py`
- Stream chunks include `usage` only on the final chunk when
  `stream_options={"include_usage": True}` is passed
- Add to kwargs; capture in stream loop; emit a final
  `{"type": "usage", "data": {...}}` event from provider
- `app/api/routes/ai.py::_event_stream` consumes the usage event,
  logs `ai.usage` structlog event, and forwards a redacted version
  to client (so frontend can show it later)

### T-2.2: Compute estimated USD cost in usage event
- Per gpt-4.1-nano: input $0.10/M, cached $0.025/M, output $0.40/M
- Helper `_estimate_cost(usage)` in openai_provider.py
- Logged alongside raw counts

### T-2.3: New endpoint `GET /api/v1/ai/usage`
- `app/api/routes/ai.py` add new route + schema
- Reads structured logs from current process? No — too coupled.
  Instead: in-memory ring buffer (per-process) of last N usage records
- Returns `{today: {requests, prompt_tokens, completion_tokens, est_usd}, total_session: {...}}`
- Acceptable per-process scope for single-tenant pet-app

## Wave 3 — Cost control

### T-3.1: Rate limit 30 → 10 req/min
- `app/api/routes/ai.py:39` — `_RATE_LIMIT = 10`

### T-3.2: Backend `min_length=3` on `/ai/suggest-category`
- `app/api/routes/ai_suggest.py:26`
- `Query(min_length=3, max_length=500)`

### T-3.3: Per-conversation tool-result cache (60s TTL)
- `app/api/routes/ai.py::_event_stream`
- Module-level `_tool_cache: dict[(conv_id, tool_name, args_hash), (result, expires_at)]`
- On `tool_call_complete`: hash args, check cache, skip the second LLM round-trip if hit
- Wait — tool result caching can't skip the **second** LLM call (we still need LLM to verbalize the cached result). What it skips is the **tool function execution**, which is local DB query — already cheap.
- **Re-evaluation:** caching tool DB results saves DB query cost (~ms), not LLM cost. The second LLM call is unavoidable.
- **Decision:** SKIP T-3.3 — false win. Document in SUMMARY.

## Wave 4 — Categorization fixes

### T-4.1: Generate embedding on category create
- `app/api/routes/categories.py::create_category`
- Add `background_tasks: BackgroundTasks` parameter
- After successful create, schedule `_refresh_embedding(cat.id, cat.name)` if ENABLE_AI_CATEGORIZATION

### T-4.2: Skip embedding refresh if name unchanged
- `app/api/routes/categories.py::update_category`
- Pre-fetch old name before `cat_svc.update_category` call
- Compare with `body.name` before scheduling task

### T-4.3: Async LRU cache on `embed_text`
- `app/ai/embedding_service.py`
- Use `async_lru` lib OR roll a simple `OrderedDict` cache (maxsize=128, no TTL)
- Cache key: lowercased trimmed text
- Latency win primarily; cost win pennies
