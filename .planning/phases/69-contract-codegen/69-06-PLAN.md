<objective>
**B5 — CI sync-guard.** Lock the contract: a script regenerates every generated
artifact from the source schema and fails if anything is out of sync (git diff
non-empty). This prevents the drift class this whole phase fixed from silently
re-appearing — a backend response change without regenerated TS/Swift types fails CI.

Work:
1. `contract/check_contract_sync.sh`: regenerate, in order, then assert no diff:
   - `make contract` (or the docker dump) → `contract/openapi.json`
   - `cd frontend && npm run gen:api` → `frontend/src/api/generated/schema.ts`
   - `python3 contract/gen_swift_dto.py` → `ios/.../Generated/GeneratedDTO.swift`
   - `git diff --exit-code -- contract/openapi.json frontend/src/api/generated/schema.ts ios/BudgetPlanner/Networking/Generated/GeneratedDTO.swift`
   On non-empty diff: print a clear message ("Generated contract types are stale —
   run `make contract && (cd frontend && npm run gen:api) && python3
   contract/gen_swift_dto.py` and commit") and `exit 1`.
2. Wire it into `.github/workflows/ci.yml` as a step/job. The web + swift generators
   only need Node + Python (stdlib) — they read the committed openapi.json. The
   backend openapi dump needs the app importable; in CI the backend job already has
   deps + Postgres (existing backend job) — run the dump there OR run the diff-only
   guard against the committed artifact in a lightweight job. Pick the approach that
   keeps CI green and still catches drift; document it.
3. `contract/README.md`: the regen pipeline + the dependency order (backend dump
   first, since web/iOS read its output) + how to fix a guard failure.
4. Add a `make contract-check` convenience target wrapping the script.

Purpose: the contract can't silently drift again — CI enforces regen-equals-committed.
Output: check_contract_sync.sh, CI step, README, make target. Guard passes on the
current (post-69-05) tree.
</objective>
