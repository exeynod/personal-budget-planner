## Tool decision (REQUIRED ‚Äî recorded here per CONTEXT)

**Decision: custom build-time script (Python) ‚Üí vanilla `Codable`.** NOT Apple
`swift-openapi-generator`.

Rationale against the 5 CONTEXT criteria:
1. **Preserve URLSession transport (decisive):** `swift-openapi-generator` forces a
   generated `Client` + `ClientTransport` abstraction and pulls `swift-openapi-runtime`
   + `swift-openapi-urlsession` (verified from Apple's docs ‚Äî it cannot emit
   models-only; it always generates the operation/transport layer). That would
   replace/wrap the hand-written `APIClient.request<T>` URLSession transport in
   `ios/.../Networking/APIClient.swift` ‚Äî exactly what CONTEXT says to avoid. The
   custom script emits plain structs that decode through the EXISTING APIClient
   `JSONDecoder` (`.convertFromSnakeCase` + the custom MSK date strategy). Transport
   untouched.
2. **XcodeGen integration:** vanilla `.swift` files drop into a `Networking/Generated`
   sources group ‚Äî covered by the existing recursive sources path. swift-openapi-generator
   wants the SPM build-tool plugin wired into the Xcode target, heavier in an
   XcodeGen-generated project.
3. **Determinism/idempotency:** our script sorts schema keys (input openapi.json is
   already sort_keys=True from B1) ‚Üí stable output for the B5 git-diff guard. The
   Apple plugin regenerates at build time into DerivedData (not committed), harder
   to git-diff-guard.
4. **Consumer churn:** vanilla structs mirror the existing handwritten Codable shape
   ‚Üí low call-site churn in 69-05. The Apple generator changes every call into
   `client.someOperation(...)` typed-throws ‚Äî large churn across all `*API.swift`.
5. **Maintainability:** the script is one Python file we own (mirrors the B2
   openapi-typescript pattern: dump-once, generate-per-stack). No third-party SPM
   dependency surface, no transport version pinning. (External libs are ALLOWED per
   owner decision ‚Äî we simply judge the custom path strictly better here.)

## Nullability contract (REQUIRED ‚Äî drives 69-05 fixture safety)

**Optionality is driven entirely by the OpenAPI `required` set of each schema:**
- A property listed in the schema's `required` array ‚Üí emit a **non-optional** Swift
  property (`let x: T`), decoded with `decode` (not `decodeIfPresent`).
- A property ABSENT from `required` (server default or genuinely optional) ‚Üí emit a
  **Swift optional** (`let x: T?`), decoded with `decodeIfPresent`.

For `CategoryRead` this resolves to: `code`, `ord`, `created_at` ‚Üí **non-optional**
(no server default in the Pydantic schema, so they are required-on-wire);
`plan_cents`, `rollover`, `paused`, `parent_id`, `tag` ‚Üí **optional** (server
defaults ‚Üí excluded from `required`). Do NOT blanket-mark all v1.0 fields required ‚Äî
only the required-set ones. This is the contract 69-05 relies on when it fixes the
test fixtures (no fixture may feed `null`/omit a now-non-optional field).
