<objective>
**B4 — Web consumer migration (read-DTO first).** Replace the handwritten web
read-DTOs with the generated schema from 69-02, starting with the most-drifted
read-DTOs: `CategoryRead`/`CategoryV10`, `Subscription*`, `Me*`, `Actual*`.

Remove the `CategoryV10` "pending schema" stubs. **Be precise about which fields
become required vs stay optional:** 69-01's B1 audit + 69-02's drift-report confirmed
`CategoryRead` now emits these fields on the wire, but ONLY `code` and `ord` are
required-on-wire (no server default) → they become **non-optional**. The fields
`plan_cents` (=0), `rollover` (="misc"), `paused` (=False), `parent_id` (=None),
`tag` (="personal") carry server defaults → they are excluded from the OpenAPI
`required` set → they STAY **optional** in the generated types. Do NOT claim all six
become required. Additionally, the handwritten `CategoryV10` is currently MISSING the
`tag` field entirely (the wire emits `tag`: "personal"|"business"|"mixed", Phase 36) —
add it (optional).

Strategy (Claude's-discretion adapter shape, per CONTEXT):
- openapi-typescript emits `components["schemas"]["X"]`. Rather than rewrite every
  call-site, add a thin `generated/adapters.ts` that aliases the generated schemas
  onto the names consumers already import (e.g.
  `export type CategoryV10 = components["schemas"]["CategoryRead"]`). Then point the
  handwritten `types.ts` read-DTO definitions at the generated source (re-export the
  alias) and DELETE the now-duplicate field lists.
- Migrate ONLY read-DTOs in this phase (CategoryRead/CategoryV10, Subscription read
  shapes, Me read shape, Actual read shape). Write-DTOs / request payloads
  (`CategoryV10UpdatePayload`, etc.) stay handwritten — deferred to 70/backlog per
  CONTEXT. But the pending-schema stubs MUST go.
- The drift-report governs required-vs-optional: type each field exactly as the
  generated schema declares it (required → non-optional; defaulted/optional →
  optional). Remove `?? default` defensive defaults at call-sites ONLY where the type
  now guarantees presence (do not change runtime behavior beyond the type — keep
  defensive defaults if they guard legacy data; document any removed default).

**Obsolete-comment removal (finding LOW-6):** The doc comments in `types.ts` /
`v10/categories.ts` claiming "as of Phase 22 the backend ... does NOT emit these
fields" / "pending Phase 22 schema update" / "Schema gap (documented in 25-03
SUMMARY)" are now **FALSE** — the Phase 25 gap-fix put these fields on the wire. The
executor MUST DELETE these comments as obsolete, NOT treat them as current truth.
(See `types.ts` ~:747, :757-789 and `v10/categories.ts` :4-26.)

Purpose: one source of truth for web read types; kill the stub drift. Zero
behavioral regression — the wire body is unchanged, only the types are now generated.
Output: adapters.ts, migrated read-DTOs in types.ts + the v10/legacy api modules,
stubs + obsolete comments removed, `tag` added, all three gates green.
