read-DTOs with the generated Codable types from 69-03, starting with the most-drifted

Remove the `CategoryV10DTO` "pending Phase 22 schema" stubs. **Be precise about
required vs optional** (per the 69-03 drift-report / OpenAPI `required` set): for
`CategoryRead`, only `code`, `ord`, and `created_at` are required-on-wire (no server
default) → they become **non-optional** (plain `decode`, no `?? default`). The fields
`planCents` (=0), `rollover` (="misc"), `paused` (=False), `parentId` (=None), `tag`
(="personal") carry server defaults → they STAY **optional** (`decodeIfPresent`). Do
NOT mark all six required. Also: the handwritten `CategoryV10DTO` is MISSING `tag`
entirely (wire emits `tag`: "personal"|"business"|"mixed", Phase 36) — the generated
type includes it (optional); adopt it.

The same required-`created_at` rule applies to the other migrated read DTOs:
`AccountRead` (accounts.py:89 `created_at: datetime`, required) and `ActualRead`
(actual.py:91 `created_at: datetime`, required) → their generated types have
`createdAt` non-optional.

## BLOCKER fix — test fixtures feeding null/omitted required fields

Six `BudgetPlannerTests/FeaturesV10/*DataTests.swift` files build JSON fixtures that
feed `null` or OMIT fields that are now NON-optional, then `try! decode` them. Once
the read DTOs typealias onto the generated structs (non-optional `created_at`/`code`/
`ord`), those decodes THROW → iOS suite RED → violates SC#3 zero-regression. This
plan MUST update every such fixture to supply a valid value matching the now-required
wire contract. Known sites (verify against the final generated types — fix exactly the
fields the drift-report marks non-optional, on whichever DTO the fixture decodes):
- `PlanDataTests.swift` (:54 `created_at: null`, :55 `code` via codeJSON→null when nil, :57 `ord: null`, :82 `created_at: null`) — Category + (likely) Actual fixtures.
- `AnalyticsDataTests.swift` (:54 `created_at: null`, :55 code→null, :57 `ord: null`, :83 `created_at: null`).
- `CategoryDetailDataTests.swift` (:39 `created_at: null` — Actual/Transaction fixture).
- `HomeDataTests.swift` (:212 `created_at: null` — Account; :263 `created_at: null` — Actual; and `makeCategory` (~:220-265) OMITS `code`/`ord` conditionally and omits `created_at` — supply valid defaults).
- `AccountsDataTests.swift` (:51 `created_at: null` — Account; :79 `created_at: null` — Actual/Transaction).

Fix rule: for each fixture that decodes a now-non-optional field, replace `null`/
omission with a valid value — `created_at` → a valid ISO8601 timestamp string the
APIClient/JSONDecoder MSK strategy accepts (e.g. `"2026-05-09T00:00:00Z"`); `code` →
a non-null slug (give `makeCategory`/codeJSON a sensible default like `"food"`);
`ord` → a `"01".."99"` CHAR(2) string (e.g. `"01"`). Do NOT touch fields that stay
optional (planCents/rollover/paused/parentId/tag may still be null/omitted). Keep the
fixture's asserted behavior identical — only the now-required fields gain valid values.

Strategy:
- The generated types live in `Networking/Generated/GeneratedDTO.swift` (namespaced
  from 69-03). Migrate read-DTOs by `typealias CategoryV10DTO = Generated.CategoryRead`
  keeping the consumer-facing name stable, or by deleting the handwritten struct and
  pointing consumers at the generated type — pick the lower-churn option per DTO and
  document it.
- The existing URLSession `APIClient.request<T: Decodable>` transport + custom
  JSONDecoder (snake_case + MSK date pin) MUST stay untouched — generated DTOs decode
  through it unchanged (that was the whole point of the B3 tool choice).
- Migrate ONLY read-DTOs. Request/Encodable payloads (e.g. `CategoryV10UpdateRequest`)
  stay handwritten — deferred to 70/backlog per CONTEXT. But the pending-schema stub
  fields + comments MUST go.
- Drop the `?? default` decode fallbacks ONLY for fields that are now required
  (code/ord/created_at); keep any fallback that guards genuinely-optional fields,
  documenting the distinction in the SUMMARY.

**Obsolete-comment removal (finding LOW-6):** the `CategoryV10DTO.swift` doc comments
claiming "as of Phase 22 the backend ... emits only the v0.x field set" / "pending
Phase 22 schema update" / "not yet on the wire" are now FALSE (Phase 25 gap-fix). The
executor MUST DELETE them as obsolete, NOT treat them as current truth.

Purpose: one source of truth for iOS read types; kill the stub drift; transport

Output: migrated read-DTOs (CategoryV10DTO, Subscription*, Me*, Transaction/Actual)
with correct required/optional split + `tag`, stubs + obsolete comments removed, six
fixture files updated, iOS build + tests green.
