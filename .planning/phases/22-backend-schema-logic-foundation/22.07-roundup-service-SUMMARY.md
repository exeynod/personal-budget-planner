# Phase 22 Plan 22.07: Roundup Service Summary

**Implemented BE-07 auto-roundup hook (DATA-MODEL §4 / CONTEXT §Area 3) end-to-end. Pure integer formula, three-gate early-exit DB hook, and additive `create_actual_v10` / `delete_actual_v10` wiring with full balance delta-accounting + cascading-child cleanup. Includes a fix-up migration `0016_v10_actual_account_id.py` that closes the schema gap deferred from plan 22.06 (so the parent transaction can carry an `account_id` for the roundup child to inherit).**

## Accomplishments

- **Migration `0016_v10_actual_account_id.py`** — adds `actual_transaction.account_id BIGINT NULL` FK to `account.id` ON DELETE RESTRICT plus the `ix_actual_account_id` index. Symmetric `downgrade()` drops index → FK → column. NULL-able for legacy v0.x rows; production rows from plan 22.07+ will carry the value.
- **ORM update in `app/db/models.py`** — `ActualTransaction.account_id: Mapped[Optional[int]] = mapped_column(BigInteger, ForeignKey("account.id", ondelete="RESTRICT"), nullable=True)` + the matching `Index("ix_actual_account_id", "account_id")` entry in `__table_args__`. The `hasattr(ActualTransaction, "account_id")` probe in `accounts.delete_account` (added in plan 22.06) now activates the txn-ref count automatically — zero coordinated edit needed.
- **`app/services/roundup.py` (241 lines)** — full BE-07 hook surface:
  - `SavingsCategoryMissingError` (Exception) — raised on enabled SavingsConfig but missing system 'savings' Category (config drift; surfaces as HTTP 500 when wired by routers in plan 22.13).
  - `compute_roundup_delta(amount_cents, base) -> int` — DATA-MODEL §4 pure integer ceiling. Operates on `abs()`, returns 0 for non-positive base or zero amount, otherwise in `(0, base)`.
  - `should_skip(delta, base) -> bool` — CONTEXT §Area 3 skip rule (delta == 0 or delta == base).
  - `get_savings_config(db, *, user_id)` / `get_savings_category(db, *, user_id)` — read helpers. Both scoped by user_id; RLS backstop.
  - `maybe_create_roundup_child(db, *, user_id, parent_txn) -> ActualTransaction | None` — three-gate hook:
    1. parent.kind != expense → None (T-22-07-01: no recursion on roundup/deposit/income).
    2. SavingsConfig absent or roundup_enabled=False → None.
    3. should_skip(delta, base) → None.
    Else: INSERT child (kind=roundup, amount=-delta, parent_txn_id, account_id, category_id=savings_cat) + UPDATE balance via `accounts.apply_balance_delta(-delta)`. No commit; caller wraps in single transaction.
- **`app/services/actual.py` extension (additive)** — appended:
  - `create_actual_v10(... account_id?, parent_txn_id?) -> tuple[ActualTransaction, ActualTransaction | None]`. Accepts 4-valued `ActualKind` enum. Relaxes `kind == cat.kind` only for the system 'savings' category (which accepts deposit/roundup by design). Validates date, resolves period, INSERTs parent, applies balance delta if account_id supplied, then calls `maybe_create_roundup_child`.
  - `delete_actual_v10(actual_id, *, user_id) -> ActualTransaction`. Snapshots parent + cascading children pre-delete, aggregates per-account positive restore deltas, deletes parent (DB FK ON DELETE CASCADE wipes children), then applies balance restore. Idempotent: raises `ActualNotFoundError` if id is gone.
  - **Legacy `create_actual` / `delete_actual` untouched** — verified by `test_legacy_create_actual_unchanged_returns_single_object`.
- **Tests `tests/services/test_roundup.py` (944 lines, 25 tests)** — far exceeds the ≥15-test acceptance bar:
  - **12 pure-function tests** (no DB needed; all pass on host without DATABASE_URL):
    - 8 from PLAN.md verbatim acceptance gate: `test_delta_100_base_10_returns_0_skip`, `test_delta_101_base_10_returns_9`, `test_delta_99_base_10_returns_1`, `test_delta_55_base_50_returns_45`, `test_delta_50_base_50_returns_0_skip`, `test_delta_negative_amount_uses_abs`, `test_delta_zero_amount_returns_0`, `test_should_skip_helper_returns_true_for_delta_eq_base`.
    - 4 extras for spec coverage: `test_should_skip_helper_returns_true_for_delta_0`, `test_should_skip_helper_returns_false_for_normal_delta`, `test_delta_base_100_spec_example` (1234/100→66), `test_delta_base_50_spec_example` (1234/50→16).
  - **13 DB-backed integration tests** covering all 14 BE-07 acceptance behaviours from PLAN.md `<behavior>`:
    - 5 negative-path gates: no SavingsConfig, disabled config, kind=income, kind=deposit, kind=roundup → child=None.
    - 6 positive-path verifications: child kind=roundup, child.amount_cents=-9 for amount=-101 base=10, child.parent_txn_id matches parent, child.account_id matches parent (T-22-07-03 cross-tenant guard), child.category_id is the system savings cat, balance reduced by delta.
    - 1 error-path: SavingsCategoryMissingError when system 'savings' Category is absent.
    - 4 end-to-end: `create_actual_v10` expense returns (parent, child) and reduces balance by 110 (parent 101 + child 9); `create_actual_v10` income returns (parent, None) and only adds parent amount; deleting parent at DB level cascades child via FK; `delete_actual_v10` restores balance for parent + child.
    - 1 backward-compat: legacy `create_actual` still returns single object.

Migration is symmetric (clean upgrade and downgrade), NULL-able to keep legacy v0.x rows valid.

### [Rule 2 - Missing critical] Relaxed `kind == cat.kind` enforcement for the system 'savings' category in `create_actual_v10`

- **Found during:** Test design phase (writing `test_create_actual_v10_expense_with_roundup_creates_child`).
- **Issue:** The system 'savings' Category is marked `kind=CategoryKind.expense` (per CONTEXT §Area 2), but it must accept `kind='roundup'` and `kind='deposit'` ActualTransactions (DATA-MODEL §4 + §3). Strict `kind == cat.kind` enforcement (as in legacy `create_actual`) would refuse the roundup child txn at the very first validator.
- **Fix:** In `create_actual_v10`, kind/cat.kind enforcement runs only for `kind in ('expense', 'income')` AND `cat.code != 'savings'`. The savings category accepts any of the 4 kinds; non-savings categories still enforce alignment for plain expense/income kinds.
- **Files modified:** `app/services/actual.py` (only inside the new `create_actual_v10` — legacy `create_actual` is untouched).
- **Commit:** `daa2e75 feat(22-07): implement roundup service + create_actual_v10 / delete_actual_v10 (BE-07)`
