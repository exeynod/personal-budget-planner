# Phase 22 Plan 03: Alembic 0014 — ActualKind Enum + Parent_txn_id + Goal/SavingsConfig + Subscription Ext Summary

## Accomplishments

### Enum migration (BE-06, CONTEXT §Area 2 verbatim)

- `ALTER TYPE categorykind RENAME TO actualkind` + `ADD VALUE IF NOT EXISTS 'roundup'` + `ADD VALUE IF NOT EXISTS 'deposit'` обёрнуты в `op.get_context().autocommit_block()` (PostgreSQL запрещает ALTER TYPE ADD VALUE внутри транзакции).
- `IF NOT EXISTS` на ADD VALUE → идемпотентность (T-22-03-03 mitigation): partial failure после rename можно ретраить.
- Создан новый type `category_kind` (`'expense'`, `'income'`) и `category.kind` переведена на него: Category остаётся 2-valued, ActualTransaction/PlannedTransaction становятся 4-valued (PlannedTransaction permissive на DB, отвергается app-layer).

### Schema additions

- **`actual_transaction.parent_txn_id`** BIGINT NULL + self-FK `fk_actual_parent_txn` ON DELETE CASCADE + partial index `ix_actual_parent_txn_id WHERE parent_txn_id IS NOT NULL`. (Composite FK на (parent_txn_id, user_id) → (id, user_id) откладывается в 0015 — там же composite UNIQUE.)
- **`subscription` extension** (BE-12):
  - `day_of_month` SMALLINT NULL + CHECK `BETWEEN 1 AND 28` (T-22-03-04 mitigation: Feb-safe)
  - `account_id` BIGINT NULL FK→account.id ON DELETE RESTRICT (нельзя удалить счёт с подписками)
  - `posted_txn_id` BIGINT NULL FK→actual_transaction.id ON DELETE SET NULL (при удалении txn регулярка просто становится не-проведённой)
  - Partial index `ix_subscription_user_day(user_id, day_of_month) WHERE day_of_month IS NOT NULL` для PLAN-list query
- **`goal` table** (BE-11): id BIGSERIAL PK, user_id FK RESTRICT, name VARCHAR(80), target_cents BIGINT NOT NULL CHECK >0, current_cents BIGINT DEFAULT 0, due DATE, created_at TIMESTAMPTZ; CHECK `char_length(name) BETWEEN 1 AND 80`; ix_goal_user_id; RLS ENABLE+FORCE (policy в 0015); GRANTs to budget_app.
- **`savings_config` table** (BE-08): user_id BIGINT PK FK→app_user ON DELETE CASCADE (1:1 design), roundup_enabled BOOL DEFAULT false, roundup_base SMALLINT DEFAULT 10 CHECK `IN (10,50,100)` (T-22-03-05 mitigation), updated_at; RLS ENABLE+FORCE; GRANTs.
- **`budget_period` extension** (BE-14 schema): misc_rollover_cents BIGINT NOT NULL DEFAULT 0, rollover_processed_at TIMESTAMPTZ NULL, partial UNIQUE INDEX `uq_period_rolled WHERE rollover_processed_at IS NOT NULL` (T-22-03-06 — DB-level idempotency complement к advisory lock).

## Threat Mitigations Applied

- **T-22-03-03** (DoS via partial autocommit_block failure): IF NOT EXISTS на ADD VALUE — retry-safe.
- **T-22-03-04** (Tampering: day_of_month > 28): CHECK constraint at DB.
- **T-22-03-05** (Tampering: roundup_base bypass): CHECK IN (10, 50, 100).
- **T-22-03-06** (Repudiation: rollover race): partial UNIQUE INDEX uq_period_rolled.
- **T-22-03-07** (Spoofing: savings_config dual config): PK=user_id единственность.

(T-22-03-01, -02 откладываются: app-layer validators в 22.05/22.12; composite FK в 0015.)

## Known Limitations

- **Downgrade не убирает enum-values 'roundup'/'deposit'**: PostgreSQL не имеет `ALTER TYPE DROP VALUE`. После downgrade type categorykind содержит 4 values (включая 'roundup', 'deposit'). Любой actual_transaction row с этими kind вызовет ошибку при последующем cast. Downgrade подходит для dev reset, не для prod rollback после данных. Задокументировано в docstring downgrade().
- **savings_config / goal данные теряются при downgrade** — ожидаемо для drop_table.
- **rollover_processed_at теряется при downgrade** — drop_column не восстановим.
