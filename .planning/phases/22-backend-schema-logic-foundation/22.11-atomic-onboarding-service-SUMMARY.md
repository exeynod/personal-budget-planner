# Phase 22 Plan 11: Atomic v1.0 Onboarding Service Summary

**One-liner:** BE-15 атомарный v1.0-онбординг через `complete_v10`
(income+accounts+9 категорий+goal?+savings_config в одной транзакции,
409 при наличии Account) и `reset_v10` (admin-wipe для повторного
онбординга), оба с upsert-идемпотентностью на `(user_id, code)`.

## Scope

Реализован BE-15 (atomic onboarding) и BE-05 (default category seeding
logic) per plan 22.11. Существующий `app/services/onboarding.py` (legacy
v0.x flow) **не тронут** — Phase 22 только добавляет v1.0-путь.

## Implementation Notes

### `complete_v10(db, *, user_id, income_cents, accounts, category_plans, goal=None, savings_config=None)`

11-шаговый flow в одной транзакции caller'а:

1. **Validators** (run BEFORE any DB write — T-22-11-08 atomicity):
   - `_validate_income`: `income_cents` ∈ (0, 10_000_000_000 коп] (= 100M ₽)
   - `_validate_category_plans`: keys ⊆ `VALID_CATEGORY_CODES` (whitelist
     T-22-11-03), каждый plan ≥ 0 и ≤ `income_cents * 4`,
     Σ plan ≤ income → иначе `PlanExceedsIncomeError`
   - `_validate_accounts`: list non-empty, bank length 1..40,
     `account.kind` ∈ `AccountKind`, ≤ 1 явный `primary=true`
2. **Conflict check** (T-22-11-01): `SELECT count(*) FROM account WHERE
   user_id = :uid` → если > 0, raise `OnboardingConflictError(409)`
3. **AppUser update**: `income_cents=...`, `onboarded_at=now()` (UTC)
4. **Accounts insert**: каждый row через ORM с `is_primary` строго
   у одного (либо явно указанный, либо первый по умолчанию)
5. **Default Categories upsert** через `_upsert_seed_categories` —
   8 строк (food/cafe/home/transit/fun/gifts/health/subs) с UPPERCASE
   russian именами per DATA-MODEL §1.3, `kind=expense`,
   `rollover=misc`, `plan_cents = category_plans[code]` (или 0)
6. **System savings Category upsert** через `_upsert_savings_category`
   — точные параметры из CONTEXT §Area 2: `code='savings',
   name='КОПИЛКА', kind=expense, ord='99', plan_cents=0,
   rollover='savings', paused=true`
7. **Optional Goal**: вызов `app.services.goals.create_goal` (валидаторы
   §6 живут там — `target > 0`, `due > today`, name 1..80)
8. **SavingsConfig**: `app.services.savings.upsert_config` (defaults
   `roundup_enabled=False, roundup_base=10` если body не передал)
9. Возвращает summary dict: `user_id, income_cents, account_ids,
   category_ids_by_code, savings_category_id, goal_id?, savings_config,
   onboarded_at`

### `reset_v10(db, *, user_id)`

Admin-only reset для re-onboarding (используется
`DELETE /api/v1/internal/onboarding/reset` в plan 22.14, защита
`X-Internal-Token` живёт на route-уровне):

1. `UPDATE subscription SET account_id = NULL` (link отключаем; rows preserved)
2. `DELETE FROM actual_transaction WHERE account_id IN (...)` (roundup
   children каскадятся через FK)
3. `DELETE FROM savings_config / goal / account` (PK-bound rows wiped)
4. `UPDATE app_user SET income_cents = NULL, onboarded_at = NULL`
5. `UPDATE category SET plan_cents = 0` (rows preserved per spec)
6. `db.expire_all()` — чтобы ORM identity-map перечитал из DB на
   следующем SELECT (raw-SQL UPDATE не обновляет Python-атрибуты)

Возвращает `{deleted_account_ids: list[int]}` для admin-аудита.

### Idempotency: re-onboarding после reset

PLAN.md spec предписывает одновременно:
- `reset_v10` сохраняет Category rows (test
  `test_reset_v10_does_not_delete_categories`)
- `complete_v10` после reset должен работать снова (test
  `test_reset_v10_allows_re_onboarding`)

Конфликт решён через `_upsert_seed_categories` /
`_upsert_savings_category`: SELECT существующих rows по
`(user_id, code)`; если row существует — UPDATE
`plan_cents/paused/is_archived` в место, чтобы не триггерить
`UniqueViolation` на `uq_category_user_code` или
`uq_category_user_id_name`. Имена/ord/code сохраняются (не
переписываются) — пользовательские кастомизации между циклами
переживают reset.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] reset_v10 raw-SQL UPDATE leaves stale ORM cache**

- **Found during:** Task 1 GREEN phase, test
  `test_reset_v10_resets_income_to_null` failed with
  `assert 10000000 is None`.
- **Issue:** `reset_v10` uses `text("UPDATE app_user SET income_cents = NULL")`
  for performance/correctness. With `expire_on_commit=False` (test session
  default), the SQLAlchemy identity-map keeps the AppUser instance with
  the old `income_cents` value. Subsequent `db.scalar(select(AppUser))`
  returns the cached instance with stale data.
- **Fix:** Added `db.expire_all()` at the end of `reset_v10` so the ORM
  refreshes affected instances from DB on next access. Also documented
  the contract in the docstring so callers know not to rely on
  stale-instance reads after reset.
- **Files modified:** `app/services/onboarding_v10.py:reset_v10`
- **Commit:** `0b27b62`

**2. [Rule 1 — Bug] Re-onboarding after reset triggers UniqueViolation**

- **Found during:** Task 1 GREEN phase, test
  `test_reset_v10_allows_re_onboarding` failed with
  `UniqueViolationError: duplicate key (user_id, name)=(75, 'ПРОДУКТЫ')`.
- **Issue:** PLAN.md says reset_v10 must keep Category rows but also
  says re-onboarding (second `complete_v10`) must succeed. The original
  implementation always INSERTed 9 new categories, which triggers the
  partial unique index `uq_category_user_id_name` because the old rows
  are still present.
- **Fix:** Refactored the seed step into `_upsert_seed_categories` and
  `_upsert_savings_category` helpers. They SELECT existing rows by
  `(user_id, code)`; if found, UPDATE the onboarding-owned fields
  (`plan_cents`, `paused`, `is_archived`). This makes `complete_v10`
  idempotent on re-onboarding without needing to delete categories.
- **Files modified:** `app/services/onboarding_v10.py:_upsert_seed_categories`,
  `_upsert_savings_category`
- **Commit:** `0b27b62`
