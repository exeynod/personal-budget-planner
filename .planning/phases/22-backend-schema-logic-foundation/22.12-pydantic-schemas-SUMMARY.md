# Phase 22 Plan 12: Pydantic v2 schemas for v1.0 API endpoints — Summary

Pydantic v2 wire contracts for every v1.0 API endpoint added in Phase 22 (accounts, goals, savings, onboarding/complete, me, subscriptions). Strict mode + extra=forbid + cross-field validators land in the schema layer so 422s surface from the route boundary without touching the service or DB.

## What was built

Six files (5 new + 1 extended), all following the same idioms:

| File | Schemas | Requirement |
|---|---|---|
| `app/api/schemas/accounts.py` | `AccountCreate`, `AccountUpdate`, `AccountRead` | BE-02 |
| `app/api/schemas/goals.py` | `GoalCreate`, `GoalUpdate`, `GoalRead` | BE-11 |
| `app/api/schemas/savings.py` | `SavingsConfigPatch`, `SavingsConfigRead`, `DepositCreate`, `SavingsSnapshotResponse` | BE-08, BE-09, BE-10 |
| `app/api/schemas/onboarding_v10.py` | `OnboardingAccountItem`, `OnboardingGoalItem`, `OnboardingSavingsConfigItem`, `OnboardingV10Body`, `OnboardingV10Response`, `OnboardingV10SavingsConfigRead` | BE-15 |
| `app/api/schemas/me_v10.py` | `MePatchV10`, `MeV10Response` | BE-01 |
| `app/api/schemas/subscriptions.py` | extended with `SubscriptionV10Update`, `SubscriptionV10Extension`, `SubscriptionPostResponse` | BE-12, BE-13 |

Common idioms:
- All request models — `ConfigDict(strict=True, extra="forbid")`
- All read models — `ConfigDict(from_attributes=True)`
- Cross-field rules — `@model_validator(mode="after")`
- Single-field rules with side-effects (date-in-future, code whitelist) — `@field_validator`
- Money bounds — `Field(gt=0, le=100_000_000_00)` for amounts; `ge=-100_000_000_00, le=100_000_000_00` for balances (matches DB CHECK constraints)

## Deviations from Plan

**1. [Style] Split single Task 1 into 6 atomic commits (one per file).**
The plan packaged all 6 file changes into a single Task. Plan-level commit guidance ("one per schema file is fine") explicitly authorised this — splitting yields a cleaner audit trail and lets the upcoming router plan (22.13) cherry-pick or revert individual schemas without disturbing siblings.

**2. [Style] Added `OnboardingV10SavingsConfigRead` as a separate class declared AFTER `OnboardingV10Response` with a forward reference + `model_rebuild()`.**
The plan code-block declared `SavingsConfigReadInline` BELOW `OnboardingV10Response` and added a parenthetical "re-order so it's defined BEFORE". Implementation chose the forward-reference pattern instead because it keeps the most semantically important class (`OnboardingV10Response`) at the top of the response section visually. Functionally identical to the plan's reorder option.

**3. [Style] Added `SubscriptionV10Update` (not in the plan's listed schemas).**
The plan's scope mentions `SubscriptionV10Update: day_of_month, account_id` (PATCH-style v1.0 extension). Implementation added it alongside `SubscriptionV10Extension` (read mixin) per the explicit scope note in the orchestrator prompt; the plan's explicit class list omitted it but the prose covered it. Not a behaviour deviation — fills the documented gap.

No Rule 1/2/3 auto-fixes applied — plan executed as written.
