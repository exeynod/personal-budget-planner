<!-- Current behavior to PRESERVE (extracted from APIClient.rawRequest switch). -->

Current hardcoded switch in APIClient.rawRequest(... skipAuth:) -> Data:
  200...299 -> return data
  401       -> onUnauthenticated?(); throw .unauthorized      (always logout)
  403       -> detail = decodeErrorDetail; if !skipAuth { onUnauthenticated?() }; throw .forbidden(detail)
  404       -> throw .notFound
  409       -> throw .conflict(detail)
  422       -> throw .unprocessable(detail)
  429       -> retryAfter header; throw .rateLimited(retryAfter:)
  default   -> throw .serverError(statusCode, detail)   (402 falls here -> no logout, swallowed by AISuggest)

429 / Retry-After SPLIT (PRE-SPECIFIED — do NOT improvise):
  KEEP 429 Retry-After handling in APIClient. It needs the `HTTPURLResponse` to read the `Retry-After` header, and the injectable policy's `map(status, data, skipAuth, decodeDetail)` signature deliberately does NOT receive the response object. So APIClient handles 429 BEFORE delegating: if statusCode == 429, parse Retry-After off the HTTPURLResponse and throw `.rateLimited(retryAfter:)` directly (exactly as today); for every OTHER status, delegate to `errorPolicy.map(...)`. The injectable policy therefore owns ONLY 401/402/403/404/409/422/default + 2xx success. This keeps the policy signature header-free and the 4 auth regression behaviors exact. Document this split in ErrorHandling.swift and APIClient. (ErrorPolicyTests do NOT need a 429 case since 429 never reaches the policy; if a 429 assertion is desired it belongs against the live APIClient, not the policy.)

Design sketch for ErrorHandling (the strategy):
  struct ErrorHandling {
      // Given the status, response data (for detail), and whether the call skipped auth,
      // produce the domain error AND whether to fire the global logout.
      // NOTE: 429 is handled upstream in APIClient (needs the response header) and never reaches here.
      var map: (_ status: Int, _ data: Data, _ skipAuth: Bool, _ decodeDetail: (Data) -> String?) -> ErrorDecision
      static let `default` = ErrorHandling(map: { ... 401/402/403/404/409/422/default + 2xx ... })
  }
  enum ErrorDecision { case success            // 2xx, return data
                       case fail(APIError, logout: Bool) }

APIClient changes:
  - Handle 429 inline FIRST (parse Retry-After off HTTPURLResponse, throw .rateLimited) — unchanged from today.
  - For all other statuses: REMOVE the inline switch; call `errorPolicy.map(status, data, skipAuth, decodeErrorDetail)`.
  - On `.fail(error, logout)`: if logout { onUnauthenticated?() }; throw error.
  - Add `var errorPolicy: ErrorHandling = .default` (instance property, injectable via init for tests).
  - `skipAuth` stays as a transport concern (whether to attach the bearer header) AND is passed into the policy so the default policy can keep the 403-skipAuth-no-logout rule. This is NOT a per-call auth flag in the suppressForbiddenHandler sense — it is a single transport parameter the policy reads; there are NO per-endpoint auth booleans.
  - PRESERVE: `default` branch maps 402 to .serverError(402, detail) with logout:false — exactly as today, so AISuggest's silent-nil contract is unchanged.

Declarative feature policy: AISuggestCategoryAPI already swallows ALL throws to nil, so it needs NO special policy — the default 402->serverError(no logout) already yields the correct silent behavior. Document this: E1 does not require AISuggest to opt into a custom policy; the win is that the per-call flag is GONE from APIClient and the matrix is now in one typed place. If a future feature needs e.g. "treat 404 as empty not error", it composes an ErrorHandling variant — show one such composable example in ErrorHandling.swift (e.g. `static func tolerating(_ statuses: Set<Int>) -> ErrorHandling`) but DO NOT wire it anywhere this phase.
