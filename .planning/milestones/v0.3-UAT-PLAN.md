10 пунктов human-verify, которые нельзя автоматизировать без live OPENAI_API_KEY
и работающего docker-compose стека. Прогоняем вместе перед закрытием milestone v0.3.

## Setup (один раз перед прогоном)

```bash
cd /Users/exy/pet_projects/tg-budget-planner
docker compose -f docker-compose.yml -f docker-compose.dev.yml up -d --build
docker compose ps      # все 5 контейнеров healthy
docker compose logs api --tail 5   # видим "INFO: Application startup complete"
```

Открыть в браузере: <http://localhost/> (TG Mini App с DEV_MODE bypass — реальный
TG WebView не нужен).

Чтобы AI-категоризация и AI-чат реально работали, в `.env` должен быть валидный
`OPENAI_API_KEY` (не `changeme`). Если ключа нет — пункты 5–10 даём ожидаемые
human-readable error сообщения (после моего фикса в `_humanize_provider_error`),
а не stacktrace; засчитываем как «графично fail» если в чате видно «AI не настроен
(проверь OPENAI_API_KEY)».

## Phase 9 — AI Assistant (4 пункта)

### UAT-1. SSE streaming визуально

**Шаги:**
1. Перейти на таб «AI» в bottom nav.
2. В пустом чате нажать suggestion chip «Каков мой баланс?».
3. Наблюдать как ответ AI появляется в bubble.

**Ожидание:**
- Bubble assistant'а появляется не сразу целиком, а **постепенно** — текст растёт по словам/токенам.
- Если AI вызывает tool (`get_period_balance`), **до** появления ответа видна purple-pulse pill «Смотрю баланс…» (`ToolUseIndicator`).
- После завершения сообщение остаётся в истории.

**Pass criteria:** видна streaming-анимация (текст не появляется одним блоком); ToolUseIndicator мигает фиолетовым.

---

### UAT-2. Suggestion chips в empty-state

**Шаги:**
1. Если в чате уже есть сообщения — нажать корзину справа сверху, удалить историю.
2. На пустом экране должны быть видны 4 чипса: «Каков мой баланс?», «Где я перерасходовал?», «Сколько потратил на еду?», «Сделай прогноз».
3. Нажать любой chip.

**Ожидание:**
- Чипсы видны над input bar (после fix `justify-content: flex-end` в `messagesEmpty`).
- Клик по чипсе **отправляет** запрос — чипсы исчезают, появляется bubble с user-сообщением, начинается streaming ответа.

**Pass criteria:** chip-клик == sendMessage(text); чипсы скрываются после первого сообщения.

---

### UAT-3. Кнопка очистки истории

**Шаги:**
1. Иметь хотя бы одно сообщение в истории (после UAT-1 уже есть).
2. Найти иконку корзины (Trash) в header'е чата справа.
3. Нажать.

**Ожидание:**
- Кнопка корзины **видна только** при `messages.length > 0` (на пустом чате её нет — правый верхний угол пустой).
- Клик отправляет `DELETE /api/v1/ai/conversation` → 204 No Content.
- Чат становится пустым, появляется empty-state с 4 чипсами.

**Pass criteria:** условная видимость кнопки + успешное удаление в БД (`docker compose exec db psql -U budget budget_db -c "SELECT count(*) FROM ai_message"` → 0).

### UAT-4. Auth защита `POST /ai/chat`

**Шаги:**
1. В отдельном терминале:
   ```bash
   curl -i -X POST http://localhost:8000/api/v1/ai/chat \
     -H "Content-Type: application/json" \
     -d '{"text": "тест"}'
   ```

**Ожидание (в DEV_MODE=false / production):** `HTTP/1.1 403 Forbidden`.

**В нашем DEV_MODE** этот тест частично проходит — ожидаем **либо** 403 если auth dependency остался, **либо** 200 (DEV bypass). В обоих случаях **не должно быть** 500 / stacktrace утечки.

**Pass criteria:** статус код корректен, тело ответа без секретов.

## Phase 10 — AI Categorization (6 пунктов)

### UAT-5. AI suggestion box появляется

**Шаги:**
1. Открыть таб «Транзакции» → под-таб «История».
2. Нажать FAB (+) внизу справа.
3. В bottom-sheet «Новая транзакция» начать вводить описание: например, «кофе» (≥3 символа).

**Ожидание:**
- Через **~500ms** после ввода появляется AI-suggestion box над/вместо `<select>` категории.
- Suggestion содержит: имя категории + confidence-bar (фиолетовая полоска `#a78bfa`).
- Если confidence ≥ 0.5 — suggestion видна; ниже — обычный select.

**Pass criteria:** debounce работает (быстрый ввод не дёргает API на каждом нажатии); suggestion появляется только при достаточной confidence.

**Если OPENAI_API_KEY невалиден:** `useAiCategorize` молча fall-back на обычный select без spam'а ошибок (silent degradation).

---

### UAT-6. Confidence bar render

**Шаги:**
1. Продолжая UAT-5 ввести более конкретное описание (e.g. «продукты пятёрочка»).
2. Дождаться suggestion.

**Ожидание:**
- Width бара пропорциональна confidence: 80% confidence → ширина бара ~80% контейнера.
- Цвет — `#a78bfa` (фиолетовый); фон бара — приглушённый.

**Pass criteria:** визуальная пропорция; нет overflow; число confidence (если выводится в процентах) согласуется с шириной.

### UAT-7. Кнопка «Сменить» возвращает обычный select

**Шаги:**
1. Иметь активный AI-suggestion box (после UAT-5/6).
2. Нажать «Сменить» в этом боксе.

**Ожидание:**
- AI-suggestion box исчезает.
- Появляется стандартный `<select>` со всеми категориями.
- Сохранение транзакции работает с выбранной из select категорией.

**Pass criteria:** UI переключается; submit формы создаёт actual_transaction с правильным `category_id`.

### UAT-8. Settings toggle отключает feature

**Шаги:**
1. Bottom nav → «Управление» → «Настройки».
2. Найти toggle «AI категоризация» (или похожий).
3. Выключить (off).
4. Вернуться в «Транзакции» → FAB → ввести описание.

**Ожидание:**
- В Settings toggle переключается, PATCH /api/v1/settings → 200.
- В ActualEditor AI-suggestion box **не появляется** даже при ≥3 символах. Сразу обычный select.
- Включить обратно → suggestion снова работает.

**Pass criteria:** глобальный gate работает в обе стороны.

---

### UAT-9. Embedding regeneration при rename категории

**Шаги:**
1. Bottom nav → «Управление» → «Категории».
2. Переименовать любую категорию (например «Продукты» → «Продукты и еда»).
3. Подтвердить.
4. В терминале посмотреть логи api:
   ```bash
   docker compose logs api --tail 20 | grep -i embedding
   ```

**Ожидание:**
- В логах должна быть запись о регенерации embedding'а для изменённой категории
  (вызов `text-embedding-3-small` для нового имени).
- В БД `category_embedding.updated_at` для этой категории должен обновиться:
  ```bash
  docker compose exec db psql -U budget budget_db -c \
    "SELECT category_id, updated_at FROM category_embedding ORDER BY updated_at DESC LIMIT 5"
  ```

**Pass criteria:** updated_at у переименованной категории — свежий timestamp; subsequent suggestion использует новый embedding.

### UAT-10. `GET /ai/suggest-category` contract

**Шаги:**
```bash
# С init data (DEV_MODE bypass — пройдёт)
curl -s "http://localhost:8000/api/v1/ai/suggest-category?q=кофе" \
  -H "X-Telegram-Init-Data: dev-mode-stub" | jq

# Низкая confidence (бессмысленный запрос)
curl -s "http://localhost:8000/api/v1/ai/suggest-category?q=qwertyuiop" \
  -H "X-Telegram-Init-Data: dev-mode-stub" | jq
```

**Ожидание (при ≥0.5 confidence):**
```json
{
  "category_id": 5,
  "name": "Кафе и рестораны",
  "confidence": 0.78
}
```

**При <0.5 confidence:**
```json
{
  "category_id": null,
  "name": null,
  "confidence": 0.23
}
```
*(или просто `category_id: null` — точная схема в `app/api/schemas/ai_suggest.py`)*

**Pass criteria:** контракт response соблюдается; threshold 0.5 виден на границе.

## Notes

- **Fail tolerance:** UAT-1, UAT-5, UAT-6, UAT-9, UAT-10 требуют валидный OPENAI_API_KEY.
  Если ключ не настроен — **graceful degradation**: вместо ответов AI видим
  human-readable «AI не настроен на сервере (проверь OPENAI_API_KEY).», вместо
  suggestion'ов в форме — обычный select. Это **не fail** UAT, а корректное
  поведение при отсутствии ключа.
- UAT-2, UAT-3, UAT-4, UAT-7, UAT-8 не требуют OPENAI_API_KEY — тестируют
  чистый UI и контракты.
