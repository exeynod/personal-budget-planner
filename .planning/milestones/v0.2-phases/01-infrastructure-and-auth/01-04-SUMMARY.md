# Phase 1 Plan 04: Auth Layer + Alembic Initial Migration Summary

**5-step HMAC-SHA256 Telegram initData validator + FastAPI dependencies (DEV_MODE bypass, OWNER_TG_ID whitelist, X-Internal-Token gate) + async-aware Alembic env.py and initial migration (8 tables, 5 enums, 4 composite indexes, 3 unique constraints).**

## Accomplishments

- `app/core/auth.py::validate_init_data` implements the 5-step HMAC-SHA256 algorithm verbatim from HLD §7.1: parse → strip hash → sorted data_check_string → HMAC("WebAppData", bot_token) → HMAC(data_check_string, secret_key) → `hmac.compare_digest` (timing-safe) → 86400s freshness check. Three exception messages match the Wave-0 RED test contract exactly: `"Missing hash"`, `"Invalid hash"`, `"auth_date expired"`.
- `app/api/dependencies.py` exposes the three primitives every downstream plan depends on: `get_db` (the symbol overridden by the async_client fixture), `get_current_user` (DEV_MODE bypass first, then HMAC validation, then OWNER_TG_ID whitelist — each failure path returns HTTP 403 with a distinct detail), and `verify_internal_token` (single-line constant-time-ish comparison; HTTP 403 on missing or wrong token).
- `app/api/middleware.py` is intentionally a docstring-only module reserved for future cross-cutting middleware (Phase 1 keeps auth in dependencies, not middleware, to allow per-router granularity per Pattern 4/5 in 01-RESEARCH).
- `alembic.ini` + `alembic/env.py` configure Alembic to run with the asyncpg driver: env.py uses `async_engine_from_config(prefix='sqlalchemy.', poolclass=NullPool)` and `asyncio.run(run_async_migrations())`, plus the `$DATABASE_URL` env var override required by `entrypoint.sh` (Phase 1 Plan 06) and `tests/test_migrations.py`.
- `alembic/versions/0001_initial.py` materialises every table, enum, index, and unique constraint required by HLD §2 and the Wave-0 `EXPECTED_TABLES` set: app_user, category, subscription, budget_period, plan_template_item, planned_transaction, actual_transaction, app_health.

## Decisions Made

- **Decision 1 — `validate_init_data` raises plain `ValueError`, not `HTTPException`.** Keeps the auth primitive framework-agnostic so it could be reused outside FastAPI (e.g. the bot or a CLI tool); the FastAPI layer in `dependencies.py` catches `ValueError` and rethrows as HTTP 403. Matches the test contract: `pytest.raises(ValueError, match="Invalid hash")`.
- **Decision 2 — `get_db` lives in `app.api.dependencies`, NOT re-exported from `app.db.session`.** The Plan 01-01 SUMMARY explicitly listed this as the symbol that `tests/conftest.py async_client` overrides via `app.dependency_overrides[get_db]`. Re-exporting from `app.db.session` would have produced a different identity and the override would silently fail.
- **Decision 3 — DEV_MODE bypass is the FIRST branch in `get_current_user`, before any header inspection.** Guarantees that `curl http://localhost:8000/api/v1/me` works in dev with `DEV_MODE=true` even when no `X-Telegram-Init-Data` header is sent. Required for D-05's stated goal of "curl-only local development".
- **Decision 4 — Manual (not autogenerated) initial migration.** Locks every column name, server default (`server_default=sa.text('now()')` for created_at; `'5'`, `'0'`, `'true'`, `'false'`, `'2'`, `'active'` for the rest), index name (e.g. `ix_subscription_active_charge`), and FK ordering (subscription before planned_transaction so the FK target exists when planned_transaction.subscription_id is created). Future migrations may use `alembic revision --autogenerate`, but the first revision is hand-curated.
- **Decision 5 — Pre-create enums + `create_type=False` on every column reuse.** `categorykind` is shared by `category.kind`, `planned_transaction.kind`, `actual_transaction.kind`; if every `sa.Enum(name="categorykind")` had `create_type=True` (the default), Alembic would emit three `CREATE TYPE categorykind` statements and fail with `psycopg2.errors.DuplicateObject` on the second. The fix mirrors the Plan 02 ORM pattern (`PgEnum(... create_type=True)` only on first usage).

### Auto-applied Rule 3 (blocking issue) deviation

**4. [Rule 3 - Blocking] Added `create_type=False` to every enum column reuse in the initial migration.**

- **Found during:** Reading the plan's Task 2 `<action>` template before writing the migration.
- **Issue:** The plan's template called `categorykind.create(op.get_bind(), checkfirst=True)` at the top (good), but inside `create_table` it wrote `sa.Enum("expense", "income", name="categorykind")` with no `create_type=False`. Default `create_type=True` would attempt to recreate the type when the column is processed, causing `psycopg2.errors.DuplicateObject: type "categorykind" already exists` against PostgreSQL on the second column referencing the same enum (`planned_transaction.kind` after `category.kind`). This would have failed `alembic upgrade head` and broken `tests/test_migrations.py` in Wave 2.
- **Fix:** Added `create_type=False` to every `sa.Enum(name=...)` call inside `op.create_table` (5 places total: `category.kind`, `subscription.cycle`, `budget_period.status`, `planned_transaction.kind`, `planned_transaction.source`, `actual_transaction.kind`, `actual_transaction.source`). The top-level `.create(... checkfirst=True)` calls are now the single source of `CREATE TYPE`. Mirrors the Plan 02 ORM pattern (`PgEnum(... create_type=True)` only on first usage in `app/db/models.py`).
- **Files modified:** `alembic/versions/0001_initial.py`
- **Verification:** `grep -c 'create_type=False' alembic/versions/0001_initial.py` → 7 (one per enum column). The migration is now topologically equivalent to what `alembic revision --autogenerate` would produce against `Base.metadata`.
- **Committed in:** `961bb2d` (Task 2).

**Total deviations:** 3 cosmetic (grep over-tight, no functional change) + 1 Rule 3 (correctness fix preventing migration crash). No Rule 4 escalation required.
