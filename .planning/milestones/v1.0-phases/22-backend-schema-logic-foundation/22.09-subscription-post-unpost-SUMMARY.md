# Phase 22 Plan 09: Subscription Post/Unpost Service (BE-13) Summary

**One-liner:** Manual provoditel'/cancel of recurrent → expense in one DB transaction with balance delta + idempotency guards.

## Objective

Implement BE-13 service-layer flows for manual «провести в факт» of a subscription:

- **post:** create `actual_transaction(kind=expense, amount=-|sub.amount|)` linked to `sub.category_id` + `sub.account_id`, apply balance delta on the account via `create_actual_v10` (BE-03 hook + roundup pass-through), store `sub.posted_txn_id = txn.id`. Idempotency: re-post → `SubscriptionAlreadyPostedError` (409).
- **unpost:** delete the linked `actual_transaction` via `delete_actual_v10` (FK ON DELETE CASCADE drops roundup children; balance restored per-account), clear `sub.posted_txn_id`. Guard: not-posted → `SubscriptionNotPostedError` (404).

Append-only — legacy `charge_subscription` / `add_subscription_to_period` / `create_subscription` / `update_subscription` / `delete_subscription` paths (SUB-01/04/05) remain untouched.

## Threat Mitigations Implemented

| Threat ID | Category | Mitigation |
|-----------|----------|------------|
| T-22-09-01 | Tampering (double post) | `posted_txn_id IS NOT NULL` check before any INSERT → `SubscriptionAlreadyPostedError` |
| T-22-09-02 | Information Disclosure (cross-tenant) | `WHERE Subscription.user_id == user_id` + RLS backstop |

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Test reads of account.balance_cents via raw SQL instead of ORM**

- **Found during:** Task 1, RED→GREEN iteration
- **Issue:** `apply_balance_delta` issues `UPDATE account SET balance_cents = balance_cents + :delta RETURNING balance_cents` — a raw SQL UPDATE that bypasses SQLAlchemy's identity map. The fixture-loaded `primary_account` ORM object stays cached at the pre-update value, and `select(Account).where(Account.id == primary_account.id)` returns the stale cached object (identity-map hit, no DB round-trip). `db_session.expire(...)` triggers a `MissingGreenlet` because async-session lazy-load is not allowed in test sync code.
- **Fix:** Use raw SQL `text("SELECT balance_cents FROM account WHERE id = :id")` in two tests (`test_post_subscription_applies_balance_delta` and `test_unpost_subscription_restores_balance`) to fetch the current balance directly. This pattern is the right way to verify changes made via `UPDATE … RETURNING` in async tests.
- **Files modified:** `tests/services/test_subscriptions_post.py`
- **Note:** This is a **test-side** workaround for an SA-async caching limitation, not a service-layer bug. Same limitation affects `tests/services/test_roundup.py::test_create_actual_v10_*_balance_*` — those tests are reproducibly broken in the same way (verified during execution). Tracking-only; out of scope per plan 22.09.

**2. [Rule 3 - Blocking] LookupError instead of new SubscriptionNotFoundError**

- **Found during:** Task 1
- **Issue:** PLAN.md `<action>` snippet references `SubscriptionNotFoundError(sub_id)` for the not-found path, but no such class exists in the codebase. The PLAN.md note acknowledges this: «Reuse existing SubscriptionNotFoundError if defined; else raise generic».
- **Fix:** Use `LookupError(f"Subscription {sub_id} not found")` — matches the existing convention in `update_subscription` and `delete_subscription` (same module). Route layer (plan 22.13) will map `LookupError` → HTTP 404 the same way it does for those endpoints.
- **Files modified:** `app/services/subscriptions.py`
- **No new exception introduced** — keeps surface minimal.
