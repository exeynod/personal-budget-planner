# Phase 22 Plan 14: Internal Onboarding Reset Endpoint Summary

Admin-only DELETE /api/v1/internal/onboarding/reset endpoint that wipes a user's v1.0-onboarding state (accounts, goal, savings_config, zeros category plans, nulls income/onboarded_at) so they can re-run /onboarding/complete in dev — gated by X-Internal-Token, mounted under internal_router, RLS-aware via set_tenant_scope before invoking reset_v10.

## What Shipped

**`app/api/routes/internal_onboarding.py`** (new)
- `internal_onboarding_router` (prefix `/onboarding`, tag `internal-onboarding`).
- Single route: `DELETE /reset` with `user_id: int = Query(gt=0)` query param.
- Defense-in-depth guard: 503 if `settings.INTERNAL_TOKEN in ("", "changeme")`.
- Calls `set_tenant_scope(db, user_id)` before `reset_v10` (RLS requires `app.current_user_id` GUC = target row's `user_id`).
- structlog audit log: `internal_onboarding_reset.ok|failed|token_not_configured` with `target_user_id` + `deleted_account_count`/`deleted_account_ids`.
- Returns `{"user_id": int, "deleted_account_ids": list[int]}`.

**`app/api/router.py`** (modified)
- Imported `internal_onboarding_router`.
- Mounted under `internal_router` after `internal_telegram_router` and `internal_bot_router`. The parent router's `Depends(verify_internal_token)` is inherited (no duplicate dep declared on the sub-router — same pattern used by `internal_bot.py` and `internal_telegram.py` to avoid double-execution per request).

## Caddyfile Review

Both `/Users/exy/pet_projects/tg-budget-planner/Caddyfile` (Let's Encrypt, lines 20–30) and `/Users/exy/pet_projects/tg-budget-planner/Caddyfile.cloudflare` (Tunnel, lines 16–21) already block external access to `/api/v1/internal/*`:

```caddy
@internal {
    path /api/v1/internal/*
}
handle @internal {
    respond "Forbidden" 403
}
```

This rule is declared *before* the `/api/*` reverse-proxy `handle` block, so external requests to the new `/api/v1/internal/onboarding/reset` path are rejected at the edge with 403 — never reaching FastAPI. The bot/worker reach internal endpoints directly over the `budget_net` docker network at `http://api:8000/...` and present `X-Internal-Token` (verified by `verify_internal_token` dependency). **No Caddyfile changes required for plan 22.14.**

## Key Implementation Details

- **Tenant-scope is required for the reset DELETEs.** Without `set_tenant_scope(db, user_id)`, the RLS policies (`tenant_isolation_account`, `tenant_isolation_goal`, `tenant_isolation_savings_config` from alembic 0015) make `current_setting('app.current_user_id', true)::bigint` return `-1` and the DELETE statements affect 0 rows. Tested implicitly by the happy path — the assertions `Account count == 0` post-reset would fail without the GUC set.
- **Categories preserved, plans zeroed.** Per `reset_v10` doc (plan 22.11), the service zeros `Category.plan_cents` rather than deleting Category rows. Rationale: PlannedTransaction / actual_transaction rows still reference categories (FK RESTRICT), and the partial unique index `uq_category_user_code` would clash on re-seed if rows were deleted then re-inserted. Verified by test 6: `len(cats) == 9` (8 default + 1 savings) after reset, all with `plan_cents == 0`.
- **Idempotency via service contract.** `reset_v10` is idempotent (DELETE WHERE user_id=:uid + UPDATE WHERE id=:uid both no-op on already-reset user). Verified by test 6's second invocation returning `deleted_account_ids: []`.
- **Audit logging.** structlog binds `target_user_id`, `deleted_account_count`, `deleted_account_ids` so a `journalctl -u api` grep for `internal_onboarding_reset.ok` reveals exactly what was wiped, by whom (well — by *anyone* with X-Internal-Token; CONTEXT §Area 3 single-tenant accept).
