# Phase 22 Plan 10: Period-End Rollover Service (BE-14) Summary

**One-liner:** Idempotent end-of-period rollover with three-layer protection (advisory lock + processed_at gate + defensive UNIQUE INDEX), wired into close_period_job so misc-remainders accumulate into next period and savings-remainders create deposit txns.

## Objective

Implement BE-14 (DATA-MODEL §3): on period close, for every non-paused, non-archived, non-system-savings category compute `remainder = max(0, plan_cents − Σ|expense.amount_cents|)`, then either:

- **`rollover='savings'`**: insert `ActualTransaction(kind=deposit, amount=-remainder, category=savings_cat, account=primary)` and `apply_balance_delta(primary, -remainder)`.
- **`rollover='misc'`**: bulk-accumulate into `next_period.misc_rollover_cents += Σremainder`.

Idempotency requirements (CONTEXT §Area 3 verbatim):

```
pg_try_advisory_lock(hashtext('close_period:' || period_id)) + LOCK_TIMEOUT 5s
inside lock: if period.rollover_processed_at IS NOT NULL: return
after success: UPDATE budget_period SET rollover_processed_at = now() WHERE id = period_id
```

Defensive `UNIQUE INDEX uq_period_rolled ON budget_period(id) WHERE rollover_processed_at IS NOT NULL` is the DB last-resort against a hypothetical double-write race that gets past the advisory lock + ORM check.

## What was built

### `app/services/rollover.py` (296 lines, NEW)

Public surface:

```python
class RolloverConfigError(Exception):
    """Raised when a savings-rollover category has remainder > 0 but the
    user has no primary account."""

async def do_period_rollover(
    session: AsyncSession,
    *,
    period_id: int,
    user_id: int,
    next_period_id: Optional[int] = None,
) -> bool:
    """Returns True on success or already-processed (no-op).
    Returns False on advisory-lock contention.
    Raises RolloverConfigError / SavingsCategoryMissingError on config drift."""
```

**Algorithm (top-to-bottom):**

1. `pg_try_advisory_xact_lock(hashtext('close_period:'||pid))` — return `False` on contention. Lock auto-releases on COMMIT/ROLLBACK.
2. SELECT period scoped by user_id; if absent return `True` (nothing to do).
3. If `period.rollover_processed_at IS NOT NULL` → log, return `True` (no-op).
4. Iterate `Category.user_id == user_id AND NOT is_archived`. Skip paused or `code == 'savings'`.
5. For each category: `Σ |amount_cents| WHERE kind=expense AND category_id=cat.id AND period_id=pid`. `remainder = max(0, plan_cents − fact)`. If 0, skip.
6. Branch on `cat.rollover`:
   - `RolloverPolicy.savings`: lazily resolve `savings_cat` (raise `SavingsCategoryMissingError` if absent) and `primary` account (raise `RolloverConfigError` if absent). INSERT `ActualTransaction(kind=deposit, amount=-remainder, category=savings_cat, account=primary, tx_date=period.period_end, source=mini_app, description=f"Остаток {cat.name} → копилку")`. Apply balance delta `apply_balance_delta(primary, -remainder)`.
   - `RolloverPolicy.misc`: accumulate `misc_total += remainder`.
7. If `misc_total > 0 AND next_period_id IS NOT NULL`: UPDATE `next_period.misc_rollover_cents += misc_total` scoped by `(id, user_id)`.
8. **Last:** `period.rollover_processed_at = now()`. Any prior exception leaves it NULL — next worker tick retries cleanly (T-22-10-05).

### `app/worker/jobs/close_period.py` (modified)

Step reordering inside `_close_period_for_user`:

| Step | Before (legacy) | After (BE-14) |
| --- | --- | --- |
| 1 | SELECT expired | SELECT expired (unchanged) |
| 2 | compute_balance | compute_balance (unchanged) |
| 3 | flip expired.status=closed | **create next period** (need next_period.id) |
| 4 | create next period | **call do_period_rollover** |
| 5 | add subscription rows | flip expired.status=closed |
| 6 | — | add subscription rows |

Rationale (documented in source): bundling new-period + rollover + status-flip + subs in one per-user tx means a rollover failure rolls back ALL of it → `rollover_processed_at` stays NULL and the next worker tick retries the whole sequence cleanly.

## Threat-model coverage

All 7 mitigations from PLAN's `<threat_model>` realised:

| Threat | Mitigation in code |
| --- | --- |
| T-22-10-01 (double rollover → duplicate deposits) | 3 layers: advisory_xact_lock + rollover_processed_at gate + UNIQUE INDEX uq_period_rolled |
| T-22-10-02 (lock starvation) | Lock keyed per period_id (not global / not per-user) — only one job per period at a time |
| T-22-10-03 (cross-tenant rollover) | All queries `WHERE user_id == :user_id`; primary defense, RLS backstop |
| T-22-10-04 (misc applied to wrong next period) | UPDATE scoped by `(id = :next_period_id AND user_id = :user_id)` — both predicates required |
| T-22-10-05 (processed_at set on failure) | `processed_at = now()` is the LAST flush; any prior raise leaves it NULL → next tick retries |
| T-22-10-06 (no primary account → repeated failures) | `RolloverConfigError` surfaces; close_period_job's per-user exception handler logs + continue |
| T-22-10-07 (account balance race) | `apply_balance_delta` is single-statement UPDATE … RETURNING — atomic, no read-modify-write |

ASVS L1: V8.2.1 (data integrity via UNIQUE INDEX + advisory lock), V8.3.1 (audit logging via structlog `rollover.done` / `rollover.skipped.lock_contention` / `rollover.already_processed` events).

## Decisions Made

- **`pg_try_advisory_xact_lock` (transaction-scope) over `pg_try_advisory_lock` (session-scope):** auto-release on COMMIT/ROLLBACK eliminates the cleanup branch + matches the PLAN spec verbatim. Session-scope locks would require an explicit `pg_advisory_unlock` in a `finally:` which adds a failure mode (unlock-on-already-released raises a notice).
- **Lazy resolution of `primary` and `savings_cat`:** the cost of those SELECTs is paid only when the first savings-rollover category with non-zero remainder is encountered. For users with all-misc categories or all-zero remainders, those queries never run.
- **Step reorder in `close_period_job`:** moved new-period creation BEFORE rollover (and BEFORE the expired.status flip). Reasoning documented in source comment + above table. Trade-off: the new period briefly exists with `starting_balance_cents = ending_balance` but expired is still `active` for ~3 SQL statements — irrelevant because the entire sequence is one transaction (no other reader sees an in-flight state).
- **Use `Σ |amount_cents|` not `Σ amount_cents`:** the codebase convention is to store expenses as positive integers (verified against `test_balance.py` + `compute_balance`'s D-02 sign rule), so the `abs()` is a no-op for current data — but it's defense-in-depth against any legacy v0.x rows that were inserted with negative amounts.
- **Reuse `SavingsCategoryMissingError`:** introducing a parallel `RolloverSavingsCategoryMissing` would split the error surface for onboarding-config-drift across two modules. The same drift surfaces from roundup hooks, manual deposits, and now rollover — one error class is correct.
