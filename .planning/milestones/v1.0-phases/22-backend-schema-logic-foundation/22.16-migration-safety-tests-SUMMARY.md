## Decisions Made

- **Schema introspection over destructive round-trip by default.** The integration runner brings the DB to head once during stack boot. Running per-revision `downgrade -> upgrade` cycles per test would require either a private DB per test (testcontainers, not currently set up at the test-fixture level) or full TRUNCATE between tests. Both add cost without commensurate signal once the schema is asserted shape-wise. The round-trip tests remain available as `MIGRATION_ROUNDTRIP=1` for dedicated CI jobs.
- **Composite-FK pair assertions check both target UNIQUE and FK constraint by name.** PostgreSQL composite FKs require a matching UNIQUE/PK target; asserting both surfaces missing-target bugs (rare but real if a future migration drops the UNIQUE without dropping the FK first).
- **Section C also asserts the SIMPLE FK was DROPPED.** Catches partial-rerun states where 0015 didn't replace `fk_actual_parent_txn` with `fk_actual_parent_txn_composite`. Explicit negative assertion is cheap insurance against alembic state drift.
- **Backfill assertions use NOT NULL + range/regex sanity checks instead of pinning specific code values.** The test DB's seed data may evolve across plans; pinning `code = 'produkty'` would be brittle. The invariant we care about — that 0013 backfill leaves no NULL/empty/invalid `code`/`ord`/`rollover` — survives any future seed change.
- **Inline fix to pre-existing `tests/test_migrations.py`.** Rule 1 (auto-fix bugs): the failing test was directly in scope (migration safety), the fix is one-line (update `EXPECTED_TABLES`), and the failure was caused by exactly the schema drift this plan is meant to catch.
