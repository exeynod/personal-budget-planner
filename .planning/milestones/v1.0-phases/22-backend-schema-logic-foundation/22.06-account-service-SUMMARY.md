## Decisions Made

See `key-decisions` in frontmatter. Highlights:

- **actual_transaction.account_id schema gap:** Spawner offered Option A (add fix-up migration `0016_v10_actual_account_id.py`) or Option B (defer). Chose **Option B** because:
  1. Wave 1 SUMMARY explicitly defers it ("recommended: add to backlog for plan 22.06").
  2. Spawner instruction `Read PLAN.md and decide based on its task list. Follow PLAN's lead.` — PLAN's task list does NOT include adding a migration; the migration recommendation appears only in the `<action>` discussion, marked "if not present (until plan 22.3 is updated)".
  3. Adding a migration mid-Wave-2 (after 0015 RLS finalize) would create a 0016 that *more naturally* belongs as part of plan 22.07 roundup (which actually needs the column for the parent.account_id inheritance pattern). Cleaner separation.
  4. The forward-compat `hasattr(ActualTransaction, "account_id")` probe in `delete_account` lets the txn-ref check activate automatically once the column lands — zero coordinated edit needed in this plan.
  
- **Dual `get_account` and `get_or_404`:** The spawner objective lists `get_account(... ) -> Account | None`, while PLAN.md acceptance criteria require `async def get_or_404`. Both are valid contracts (None-on-miss vs. raise-on-miss); implemented both rather than picking one. Routers will use `get_or_404` for HTTP 404 mapping; service-to-service callers (e.g., onboarding probe) can use `get_account` to avoid try/except boilerplate.

- **Sole-primary orphan guard in `delete_account`:** Plan 22.13 will eventually need to migrate transactions/subscriptions before deleting their account. For the service layer, refusing to delete the sole primary when other accounts exist preserves the onboarding invariant (every user must always have one primary). Single-account deletion is permitted because auto-primary fires on the next `create_account`. ValueError (not AccountHasTxnsError) — distinct error class so routers can map the two failures to different HTTP codes (409 for refs, 422 for orphan-primary).

## Known Side Effects (for plans 22.07+)

### Schema gap: `actual_transaction.account_id`

`apply_balance_delta` is implemented and tested for direct `account_id` calls. However, in plans 22.07 (roundup) and 22.13 (POST /api/v1/actual router), the txn services need to call `apply_balance_delta(... account_id=parent.account_id ...)` where `parent` is an ActualTransaction row. **The column does not exist yet.**

**Two paths forward** (decide in plan 22.07 planning):

1. **Path A (recommended):** Plan 22.07 adds migration `0016_v10_actual_account_id.py` adding `actual_transaction.account_id BIGINT NULL FK account.id ON DELETE RESTRICT + ix_actual_account_id index`. Updates ORM in plan 22.07's task. Adds the column to `ActualTransaction.__init__` callers in plan 22.13's router. The `hasattr` probe in `delete_account` automatically picks up the new ref.

2. **Path B:** Plan 22.07 stores `account_id` only on the parent payload but resolves it transitively via `subscription.account_id` (for sub-driven txns) or `primary_account.id` (for manual txns). Slower (extra JOIN) and more fragile — not recommended.

### Pre-existing api/bot/worker import errors (carry-over from Wave 1)

`docker logs tg-budget-planner-api-1` shows `ImportError: cannot import name 'PlanTemplateItem'` from `app.services.planned`. This is the documented Wave 1 known side effect (plan 22.05 SUMMARY) and is in plan 22.13's scope to resolve. Plan 22.06 does NOT fix it — out of scope per the spawner instruction "DO NOT fix them in this plan unless they directly block your work". My tests never imported `app.services.planned`, so the broken module did not block.
